# An FPGA Implementation of Displacement Vector Search for Intra Pattern Copy in JPEG XS

**会议**: CVPR 2025
**arXiv**: [2603.10671](https://arxiv.org/abs/2603.10671)
**代码**: 待确认
**领域**: 模型压缩 / 硬件加速 / 视频编码
**关键词**: FPGA实现, JPEG XS, 帧内模式复制, 位移向量搜索, 硬件加速

## 一句话总结

首次提出JPEG XS帧内模式复制(IPC)中位移向量(DV)搜索模块的FPGA架构实现，采用四级流水线设计和优化的存储组织方式，在Xilinx Artix-7上实现38.3 Mpixels/s吞吐量和277 mW功耗，为IPC实际硬件部署和ASIC转化奠定基础。

## 研究背景与动机

### 领域现状
JPEG XS是JPEG委员会开发的低延迟、低复杂度图像压缩标准，面向远程桌面、KVM应用等场景。为提升JPEG XS在屏幕内容上的编码效率，已有多种技术被开发，包括时间差分编码(TDC)和帧内模式复制(IPC)。

### 现有痛点
- IPC通过小波域帧内补偿预测减少图像结构冗余，能显著提升屏幕内容的BD-PSNR
- 但DV搜索需遍历所有候选模式并求解最优预测偏移，计算量大，是IPC中最耗计算资源、最占资源、最敏感延迟的模块
- 现有H.264/HEVC运动估计和帧内预测的FPGA/ASIC实现均不适用于JPEG XS的低延迟低复杂度框架

### 核心矛盾
IPC的DV搜索在算法层面带来显著编码增益，但其计算密集性成为实际硬件部署的关键瓶颈，阻碍了IPC在实时硬件系统中的实用化。

### 本文要解决什么
设计高效的FPGA架构实现DV搜索模块，使IPC能够在实际硬件上部署，并为后续ASIC实现提供参考。

### 切入角度
从JPEG XS IPC的计算特性和数据复用模式出发，设计针对性的流水线架构和存储优化方案。

### 核心idea
四级流水线DV比较架构 + IPC Group对齐的外部存储组织，利用IPC的分组计算特性和数据内在复用模式来优化性能。

## 方法详解

### 整体框架
DV搜索架构包含两个主引擎：**残差计算引擎**（从DRAM获取IPC Unit数据并计算残差）和**DV比较引擎**（基于残差比特平面开销选择最优DV）。系统通过流水线设计平衡了吞吐量和延迟。

### 关键设计1：残差计算引擎

- **做什么**：从不同子带检索原始和重建小波系数块，并计算块级残差
- **核心思路**：通过CMD模块从DRAM获取系数，经FIFO缓冲对齐后，由SIG_MAG_SUB模块执行符号-幅度格式的有符号减法。实现4条并行减法路径处理不同符号组合
- **设计动机**：IPC的系数来自不同子带且分散存储，需要通过FIFO和MUX确保数据按组对齐同步读取

### 关键设计2：四级流水线DV比较引擎

- **做什么**：评估每个残差IPC Unit的比特开销并搜索最优匹配DV
- **核心思路**：
  - Stage 0：输入残差系数和DV，生成BandIdx/GrpSize/UnitWidth等配置参数
  - Stage 1：GetOrMask对组内残差做按位OR掩码计算
  - Stage 2：CalGCLI基于OR结果计算GCLI比特开销
  - Stage 3：Compare模块比较当前与历史最小开销，更新最优DV
- **设计动机**：系统化的流水线设计使能并行DV评估、低延迟处理，支持可扩展的块级处理

### 关键设计3：IPC Group对齐的外部存储组织

- **做什么**：优化DRAM中小波系数的存储布局
- **核心思路**：
  - Method 0 (基线)：按Precinct顺序存储，简单线性寻址但IPC分组访问效率低
  - Method 1 (提出)：按IPC Group和Unit组织系数，同一组内IPC Unit顺序存储，每组包含所有子带块
- **设计动机**：Method 0需按组/单元/子带索引逐个定位分散的系数块，增加控制复杂度降低存储吞吐；Method 1允许单一基地址加固定偏移即可加载整个IPC Unit，支持突发式存储访问
- 片上TLB RAM存储不同组中IPC Unit各系数块的长度信息

### 损失函数/训练策略
不适用（硬件设计论文，无训练过程）。目标是在保持与IPC参考软件一致的率失真性能前提下，最大化硬件吞吐量并最小化资源占用。

## 亮点与洞察

1. **首创性**：首次为JPEG XS IPC框架中的DV搜索提出FPGA实现方案
2. **存储组织创新**：从Precinct对齐到IPC Group对齐的存储重组，利用了IPC的分组计算特性，是针对领域特定的巧妙优化
3. **实用导向**：设计以ASIC部署为最终目标，FPGA作为验证平台，路线清晰
4. **片上TLB设计**：通过TLB RAM存储变长块大小信息，优雅处理不同IPC Group中系数块大小不一致的问题

## 局限性 / 可改进方向

1. **仅实现DV搜索一个模块**：未涵盖IPC完整预测环路（模式补偿、模式选择等），离IPC全硬件实现仍有距离
2. **FPGA平台较小**：使用Artix-7 (XC7A35T)，资源有限，可能限制了设计的优化空间
3. **吞吐量有限**：38.3 Mpixels/s对于4K@60fps（约497 Mpixels/s）等高分辨率实时应用仍有差距
4. **未报告与其他编码标准硬件实现的对比**：缺乏与H.264/HEVC帧内预测硬件的系统性对比
5. **BRAM增加的权衡**：Method 1以增加BRAM为代价换取吞吐量和逻辑资源优化，在BRAM紧张的场景可能受限
6. **未探索更高并行度设计**：如多DV并行搜索或多组并行处理

## 相关工作与启发

- **与H.264/HEVC运动估计硬件的区别**：传统视频编码硬件基于像素域块匹配和SAD/SATD代价评估，而IPC工作在小波域、使用分组频域预测流
- **与JPEG XS TDC的互补关系**：TDC做帧间时间预测，IPC做帧内空间预测，二者可组合使用
- **对小波域压缩硬件的启发**：IPC Group对齐存储的思路可推广到其他小波域压缩算法的硬件实现
- **ASIC部署前景**：FPGA验证为低功耗ASIC设计提供了资源预估和架构参考
