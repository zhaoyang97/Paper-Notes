# ArgHiTZ at ArchEHR-QA 2025: A Two-Step Divide and Conquer Approach to Patient Question Answering for Top Factuality

**会议**: ACL 2025  
**arXiv**: [2506.12886](https://arxiv.org/abs/2506.12886)  
**代码**: https://github.com/hitz-zentroa/ArchEHR-ArgHiTZ  
**领域**: NLP理解 / 医学NLP  
**关键词**: 患者问答, 电子健康记录, 共享任务, 两阶段方法, 事实性

## 一句话总结
在 ArchEHR-QA 2025 共享任务中提出两阶段"分治"方法：先用重排序模型从电子健康记录中提取关键句子，再用小型医学 LLM 生成回复，在不使用外部知识的情况下取得事实性排名第一、总分第 8/30 的成绩。

## 研究背景与动机
1. **领域现状**：医疗机构的在线患者门户中，患者消息量不断增加，给临床医生带来巨大负担。ArchEHR-QA 2025 任务要求根据患者的电子健康记录 (EHR) 自动回答患者提出的健康问题。
2. **现有痛点**：直接用端到端 LLM 生成回复，需要模型同时完成句子选择和回复生成两个子任务，准确率不够高；大模型（70B）并不比小模型效果好，但成本高很多。
3. **核心矛盾**：在仅有 20 条开发集样本、无训练数据的条件下，如何让系统既能准确识别 EHR 中的关键证据句，又能生成高质量的患者回复。
4. **本文要解决什么？** 用任务分解策略分别优化证据选择和回复生成，在低资源场景下提升事实性和整体质量。
5. **切入角度**：将任务拆分为两步——第一步用最适合的方法（重排序器/提示）选关键句，第二步用医学 LLM 基于选出的句子生成回复，比端到端更可控。
6. **核心 idea 一句话**：关键句提取用相似度重排序器，回复生成用小型医学 LLM，任务分解让每个子任务用最适合的方法。

## 方法详解

### 整体框架
输入：患者关注事项 + 技术问题重述 + 带句子 ID 的 EHR 全文
输出：≤75 词的回复段落，每句引用 EHR 句子 ID

三种方案对比：
1. **端到端基线**：直接提示 LLM 一步生成回复+引用
2. **两阶段-提示**：第一步用 LLM 提示提取关键句 → 第二步生成回复
3. **两阶段-重排序**：第一步用重排序模型提取关键句 → 第二步生成回复

### 关键设计

1. **端到端基线 (End-to-End)**:
   - 做什么：用 role prompting + CoT + 1-shot 示例，让 LLM 直接输出带引用的回复
   - 核心思路：输入患者叙述、技术问题、带 ID 的 EHR，模型作为医生角色一步生成
   - 发现：8B 的 Aloe 模型优于 70B 的 Llama 3.3 和 Aloe 70B，说明大模型不一定更好

2. **两阶段-提示方法 (Two-Step Prompting)**:
   - 做什么：第一步用 LLM 提示提取关键句列表；第二步基于关键句生成回复
   - 两种提取策略：(a) 提取列表——让模型一次性输出所有关键句 ID；(b) 逐句判断——对每句独立判断是否关键（Yes/No）
   - 尝试多种提示策略：基础提示、角色提示、CoT、one-shot/few-shot
   - 最优组合：Aloe 8B + 角色提示，Strict F1=0.50

3. **两阶段-重排序方法 (Two-Step Re-ranker)**（最佳方案）:
   - 做什么：第一步用相似度重排序模型对 EHR 句子按与问题的相关性排序，阈值筛选关键句
   - 核心思路：将患者叙述+技术问题作为 query，EHR 各句子作为 document，用 reranker 打分，通过 ROC 曲线 + Youden 指数确定最优阈值
   - 测试了 Jina Reranker、BAAI BGE、Alibaba GTE 三个模型，Jina 最优 (F1=0.535)
   - 第二步同样用 Aloe 8B 生成回复，再后处理添加引用

### 后处理
- 控制回复在 75 词以内，按句子选择截断
- 基于相似度匹配为每句回复添加最相关的 EHR 句子 ID 引用
- 排除标题类句子，限制每句最大引用数

## 实验关键数据

### 主实验

| 方法 | Dev Overall | Dev Factuality | Test Overall | Test Factuality |
|------|-------------|----------------|--------------|-----------------|
| End-to-End (Aloe 8B) | 0.388 | 0.464 | 0.367 | 0.408 |
| Two-Step Prompting | 0.385 | 0.504 | 0.366 | 0.452 |
| **Two-Step Re-Ranker** | **0.421** | **0.558** | **0.440** | **0.605** |
| Baseline (Llama 3.3 70B) | 0.359 | - | 0.307 | - |

### 消融：重排序模型对比

| 模型 | Strict Prec. | Strict Rec. | Strict F1 |
|------|-------------|-------------|-----------|
| Jina Reranker | 0.427 | **0.717** | **0.535** |
| Alibaba GTE | 0.422 | 0.681 | 0.521 |
| BAAI BGE | **0.507** | 0.507 | 0.507 |

### 关键发现
- 两阶段-重排序方法全面最优，Test 上 Overall=0.44（第8/30名），**Factuality 排名第一**
- 任务分解本身不一定有效（提示法两阶段 vs 端到端差异不大），关键是**每个子任务用对方法**
- 小模型 Aloe 8B 全面优于 Llama 3.3 70B 和 Aloe 70B，说明医学领域微调比规模更重要
- 重排序器比 LLM 提示更适合做证据句选择任务
- Relevance 分数仍有提升空间，说明回复生成质量还可以改进

## 亮点与洞察
- **子任务适配方法**：不是简单分解任务就好，而是要为每个子任务选择最合适的方法——证据选择用检索模型，回复生成用语言模型
- **小模型优于大模型**：在需要医学专业知识+摘要+改写的复杂任务中，领域微调的 8B 模型胜过通用 70B 模型，对资源受限场景有价值
- **无需外部知识**：仅利用任务提供的 EHR 数据，不依赖医学知识库，仍能取得事实性第一

## 局限性 / 可改进方向
- 仅有 20 条开发集样本，阈值在如此小的数据上调参可能过拟合
- 未尝试 fine-tuning（无训练数据限制），也未使用 RAG 等外部知识
- Relevance 分数较低，回复的可读性和回应质量还有提升空间
- 可尝试 text-to-text 模型（如 T5）进行回复改写
- 引用匹配基于简单相似度，可能不够精确

## 相关工作与启发
- **vs 端到端 LLM**：直接端到端生成在 Factuality 上弱于两阶段方法，说明将证据选择与回复生成分离有助于提高准确性
- **vs RAG 架构**：两阶段-重排序方法本质上是 RAG 的变体，但不使用外部知识库，仅在给定 EHR 内检索
- 任务分解+子任务适配的思路可以推广到其他复杂 NLP 任务
