# A Practical Approach for Building Production-Grade Conversational Agents with Workflow Graphs

**会议**: ACL 2025  
**arXiv**: [2505.23006](https://arxiv.org/abs/2505.23006)  
**代码**: 待确认  
**领域**: 对话系统 / 工业应用  
**关键词**: 对话Agent, 工作流图, DAG, 电商Agent, LLM微调, 合规约束, 格式控制  

## 一句话总结

提出基于有向无环图（DAG）工作流的混合对话 Agent 框架——每个图节点有独立系统提示/工具/执行规则以处理特定场景约束。结合原型 Agent 数据收集、状态感知的响应掩码微调策略，在 Kakao 移动电商场景中任务准确率提升 52%、格式遵从度提升 50%，超越 GPT-4o。

## 背景与动机

LLM 在对话 Agent 中展现出强大能力，但在工业级部署中面临三大挑战：

1. **合规遵从**：电商场景需严格遵守业务规则（如禁止向未成年人推荐烟酒），但 LLM 的概率生成本质导致偶尔违规
2. **格式控制**：移动端需特定输出格式（长度限制、emoji 项目符号），必须避免夸大广告并正确引用来源
3. **复杂约束下的 prompt 膨胀**：将所有规则写入系统 prompt 导致 prompt 过长，降低延迟和准确率

## 核心问题

如何构建既保持灵活对话能力又严格遵守场景特定业务约束的生产级对话 Agent？

## 方法详解

### 工作流图设计

- Agent 建模为 DAG：节点=状态（LLM 调用/工具调用），边=状态转移
- 每个 LLM 节点有独立系统 prompt（仅含该状态的约束）+ modify_history 子程序（操控对话历史防止幻觉）
- 工具节点有输入/输出 schema，使用 constrained decoding 保证 JSON 格式
- 通用聊天节点作为入口，路由到任务特定节点

### 数据收集

- 用 GPT-4o 构建原型 Agent → 人工标注员与原型交互收集对话 → 标注员修正错误回复
- 静态类型检查器辅助修正 JSON 格式问题

### 响应掩码微调

- 关键挑战：同一对话中不同回复来自不同节点（不同 prompt），标准多轮训练会引入冲突监督信号
- 解决方案：对每个节点 v 的训练数据 $(s_v, x_1, o_1, ..., x_n, o_n)$，仅在属于节点 v 生成的回复上计算损失，掩码其他节点的回复
- 这防止模型在错误 prompt 约束下学习不匹配的回复风格

## 实验关键数据

| 指标 | 基线 (GPT-4o) | 本方法 (微调) | 提升 |
|------|-------------|------------|------|
| 任务准确率 | ~50% | ~76% | **+52%** |
| 格式遵从度 | ~50% | ~75% | **+50%** |

- 超越 GPT-4o 在电商场景的表现
- 已在 Kakao 移动聊天平台实际部署

## 亮点

- **图结构分散约束**：每个节点只处理自己的约束，避免 prompt 膨胀
- **响应掩码微调巧妙**：解决了图-LLM 耦合系统中训练信号冲突的关键问题
- **完整的工业级部署报告**：从设计到数据收集到训练到部署，提供可复现的方法论
- **modify_history 防幻觉**：限制 LLM 只看相关上下文

## 局限性 / 可改进方向

- **图结构需人工设计**：DAG 拓扑依赖领域专家手工定义
- **仅在电商场景验证**：其他领域（医疗、金融）未测试
- **数据收集成本高**：需人工标注员与原型 Agent 交互并修正
- **DAG 限制**：有向无环图可能无法处理需要回溯的复杂对话流

## 与相关工作的对比

- **vs 纯 LLM Agent（ReAct/ToolLLM）**：灵活但无法保证合规；本方法用图结构强制约束执行路径
- **vs Rasa/Talkamatic 规则系统**：可靠但缺乏灵活性；本方法在图节点内保留 LLM 的生成灵活性
- **vs MARCO 护栏方法**：MARCO 依赖重试修正，增加延迟；本方法通过图结构和微调一次性做对

## 启发与关联

- DAG + 独立 prompt per 节点 + 响应掩码微调是一套可推广的工业级 Agent 构建方法论
- "把约束分散到图节点而非堆在一个 prompt"是处理复杂业务规则的通用思路
- 状态依赖的训练数据构建问题在所有多状态/多角色 Agent 中都会出现
