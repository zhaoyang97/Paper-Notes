# ScaleQuest: Unleashing LLM Reasoning Capability via Scalable Question Synthesis from Scratch

**会议**: ACL 2025  
**arXiv**: [2410.18693](https://arxiv.org/abs/2410.18693)  
**代码**: https://scalequest.github.io  
**领域**: LLM推理  
**关键词**: math reasoning, question synthesis, QFT, QPO, data scaling

## 一句话总结
提出 ScaleQuest，通过 Question Fine-Tuning (QFT) + Question Preference Optimization (QPO) 两阶段解锁 7B 模型的数学题生成能力，从零合成 100 万高质量问题-解答对，训练效果超越所有开源数据集并展现持续的数据扩展潜力。

## 研究背景与动机

1. **领域现状**：提升 LLM 数学推理需要大规模高质量数据。现有方法依赖种子数据改写（MetaMath）或知识图谱引导（NuminaMath），多样性受限且依赖 GPT-4 成本高。
2. **现有痛点**：(1) 问题驱动方法（改写/evol-instruct）多样性低——生成的问题与种子高度相似。(2) 知识驱动方法虽更多样但仍依赖强模型。(3) 直接用解题模型生成问题效果差（Magpie 方式不适用于推理数据）。
3. **核心矛盾**：解题模型的训练损失只在回答上计算，问题生成能力没有被显式激活。
4. **本文要解决什么？** 用轻量 7B 模型低成本合成大规模多样化数学推理数据。

## 方法详解

### 关键设计

1. **QFT (Question Fine-Tuning)**: 用 ~15K 问题（无解答）微调解题模型，在问题末尾加 EOS token 让模型学会"只生成问题"。验证了 QFT 激活的是生成能力而非记忆——用 GSM8K 和 MATH 训练的模型生成的问题分布趋同。

2. **QPO (Question Preference Optimization)**: 用外部 LLM 优化 10K 问题的可解性和难度，构造偏好对做 DPO 训练。每次随机选一个优化方向（可解性 or 难度），避免同时优化的冲突。

3. **过滤+响应生成**: 语言过滤（去非英文 ~20%）+ 可解性过滤（Qwen2-Math 判断）+ 难度采样（基于 fail rate 的难度评分器）+ Best-of-5 奖励模型选择最优解答。

## 实验关键数据

| 数据集 | GSM8K | MATH | OlympiadBench | 数据量 |
|------|:---:|:---:|:---:|:---:|
| MetaMath | 基线 | 基线 | 基线 | 395K |
| DART-Math | 可比 | 可比 | - | 585K |
| NuminaMath | 中等 | 中等 | - | 860K |
| **ScaleQuest** | **最优** | **最优** | **持续提升** | **1M** |

**关键结果**: 同等数据量下超越所有开源数据集。数据量从 100K→1M 性能持续提升，未见饱和。代码推理任务也有显著泛化。

## 亮点与洞察
- **"从解题到出题"的范式转变**：QFT+QPO 将解题模型变成出题模型，只需约 15K 种子问题（无解答）即可。
- **数据扩展不饱和**：100K→1M 持续提升，暗示可以继续扩展到更大规模。

## 亮点与洞察
- **"从解题到出题"的范式转变**：QFT+QPO 将解题模型变成出题模型，只需约 15K 种子问题（无解答）。
- **数据扩展不饱和**：100K→1M 持续提升且无收敛迹象——暗示可继续扩展到更大规模。
- **QPO 的偏好优化巧妙**：每条只随机选一个优化方向（可解性 or 难度），避免同时优化的冲突。
- **代码推理也有效**：方法从数学泛化到代码推理任务。

## 局限性 / 可改进方向
- QPO 数据的外部 LLM 优化可能引入偏差。难度采样的阈值需经验设置。仅测试 7B 模型。
