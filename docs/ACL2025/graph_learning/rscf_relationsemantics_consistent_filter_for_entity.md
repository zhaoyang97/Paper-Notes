# RSCF: Relation-Semantics Consistent Filter for Entity Embedding of Knowledge Graph

**会议**: ACL 2025  
**arXiv**: [2505.20813](https://arxiv.org/abs/2505.20813)  
**代码**: 有  
**领域**: LLM/NLP  
**关键词**: KGE, entity transformation, relation semantics, consistency, plug-in

## 一句话总结
提出 RSCF 插件式 KGE 方法，通过共享仿射变换 + 根植实体变换 + 归一化三特征确保"语义相似的关系产生相似的实体变换"（关系语义一致性），在距离模型和张量分解模型上均显著超越 SOTA，并从理论和实验上验证了一致性保持率。

## 研究背景与动机

1. **领域现状**：KGE 通过将实体和关系映射到低维向量空间来进行知识图谱补全。为处理 1-N/N-1/N-N 等复杂关系模式，实体变换模型（ETM）对每个关系生成不同的实体嵌入（如 SFBR 用关系特定的 $W_r$ 矩阵变换实体）。
2. **现有痛点**：(1) **变换断裂问题**：每个关系的变换参数 $W_r$ 互相独立，语义相似的关系可能学到完全不同的变换和实体嵌入。稀疏 KG 中有限的实体观测使这种断裂更严重。(2) **嵌入过度集中**：SFBR + DURA 正则化导致实体嵌入过度集中，生成跨关系不可区分的评分分布——"所有关系看起来都一样"。
3. **核心矛盾**：如何既利用关系各自的特异性（区分不同关系），又保持语义相似关系的一致性（相似关系应有相似的变换）？
4. **核心 idea**：用共享的仿射变换连接所有关系嵌入到实体变换空间，确保"输入相似→输出相似"的一致性映射。

## 方法详解

### 整体框架
实体变换公式：$\mathbf{e_r} = (\text{N}_p(\mathbf{rA_1}) + \mathbf{1}) \otimes \mathbf{e}$，其中 $\mathbf{A_1}$ 是所有关系共享的仿射变换矩阵，$\text{N}_p$ 是 $L_p$ 归一化，$\otimes$ 是逐元素乘积。

### 关键设计

1. **共享仿射变换 (ⓐ)**:
   - 所有关系共享同一个线性变换 $\mathbf{A_1} \in \mathbb{R}^{n \times n}$，将关系嵌入映射为实体变换向量。
   - 仿射变换保持平行性和距离比例→语义距离近的关系嵌入会映射到距离近的变换向量。
   - Monte Carlo 模拟验证：一致性保持率在直线情况下 100%，非直线情况下 72.8%~99.4%。

2. **根植实体变换 (ⓒ)**:
   - 变换结果 $\mathbf{e_r}$ = 原始嵌入 $\mathbf{e}$ + 关系特异的"变化量"，而非完全替换。
   - "加一"操作：$\text{N}_p(\mathbf{rA_1}) + \mathbf{1}$ 确保变换是原始嵌入的"微调"而非"重置"。
   - 设计动机：降低变换所需的表达力（只需学变化量而非全新位置），使共享仿射变换的表达力损失可控。

3. **归一化 (ⓑ)**:
   - $\text{N}_p(\mathbf{rA}) = \frac{\mathbf{rA}}{\|\mathbf{rA}\|_p}$ 防止变化量尺度无限缩小或放大。
   - 在安全范围内约束变换量，保护根植变换的语义保留能力。

4. **关系变换 (RT) + 关系预测 (RP)**:
   - RT：用同样的 RSCF 机制对关系嵌入做实体特异变换 $\mathbf{r_{ht}} = \psi'(h) \otimes \psi'(t) \otimes \mathbf{r}$，捕获子关系语义。
   - RP：训练目标中加入关系预测损失 $\phi(\mathbf{r}|\mathbf{h},\mathbf{t})$，增强关系嵌入的语义聚类效果。

## 亮点与洞察
- **"关系语义一致性"概念**的形式化定义（任意关系对的距离关系在变换后保持）是核心贡献，将模糊的直觉转化为可验证的数学性质。
- **Monte Carlo 验证**是一个巧妙的理论分析方法——在随机生成的嵌入上验证一致性保持概率。
- **极简设计**：整个 RSCF 只增加 3 个共享矩阵 $A_1, A_2, A_3$（各 $n \times n$），参数增加极少。

## 局限性 / 可改进方向
- 共享仿射变换的表达力上界受限——极不相似的关系可能需要差异更大的变换。
- 仅在标准 KGC 数据集测试，在超大规模 KG 上的可扩展性未验证。
- RP 和 RT 模块是否可以更高效地实现。
