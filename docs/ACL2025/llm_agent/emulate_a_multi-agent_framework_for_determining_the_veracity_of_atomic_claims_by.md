# EMULATE: A Multi-Agent Framework for Determining the Veracity of Atomic Claims by Emulating Human Actions

**会议**: ACL 2025  
**arXiv**: [2505.16576](https://arxiv.org/abs/2505.16576)  
**代码**: https://github.com/qqqube/EMULATE (有)  
**领域**: LLM Agent  
**关键词**: 多智能体, 事实核查, 声明验证, 迭代检索, 网络搜索

## 一句话总结
提出 EMULATE 多智能体事实核查框架，通过 7 个专职 LLM agent 模拟人类验证声明的完整行为链（搜索→排序→内容评估→证据充分性判断→分类），在三个事实核查 benchmark 上的 Macro-F1 和 Weighted-F1 均超越现有方法。

## 研究背景与动机

1. **领域现状**：自动事实核查系统通常采用"分解-验证"范式：将文本拆为原子声明，然后逐条验证。验证过程一般是"检索证据→LLM 分类"的两步流水线。
2. **现有痛点**：(a) 传统方法一次检索所有证据再分类，与人类的迭代式验证行为不一致；(b) 搜索结果未经可信度/相关度筛选，噪声多；(c) 网页内容可能不自包含（需要额外上下文才能理解），直接用作证据会误导分类。
3. **核心矛盾**：单轮检索+直接分类的范式遗漏了人类验证过程中的关键步骤——评估来源可信度、判断内容完整性、决定是否需要继续搜索。
4. **本文要解决什么？** 设计一个更接近人类行为的多智能体框架，让每个 agent 负责验证流程中的一个小步骤。
5. **切入角度**：分析人类验证声明的完整行为链（搜索→选择链接→阅读判断→决定下一步），为每个步骤设计专门的 LLM agent。
6. **核心 idea 一句话**：用 7 个专职 LLM agent 分别处理搜索、排序、完整性判断、有用性判断、充分性判断、分类等子任务，逐步构建高质量证据集进行事实核查。

## 方法详解

### 整体框架
输入是一条原子声明(atomic claim)，输出是 True/False 标签。流程：(1) InitialQueryGen 生成初始搜索查询；(2) 搜索引擎返回结果，SearchRank 按可信度和相关度排序；(3) 遍历搜索结果，SelfContainedCheck 判断内容是否可理解，DetHelpful 判断是否提供新信息；(4) 持续迭代，SufficientEvidence 判断证据是否足够；(5) 不够则 AdditionalQueryGen 生成新查询继续搜索；(6) 最终 Classifier 基于证据集做分类。

### 关键设计

1. **7 个专职 Agent 设计**:
   - **InitialQueryGen**: 根据声明生成初始搜索查询列表
   - **SearchRank**: 根据 URL 可信度和标题/摘要相关度对搜索结果排序，确保优先处理高质量来源
   - **SelfContainedCheck**: 判断网页内容是否自包含（可独立理解或结合已有证据可理解）。不自包含的结果暂存，等证据集丰富后再处理
   - **DetHelpful**: 判断可理解的搜索结果是否提供了证据集中尚未包含的新信息
   - **SufficientEvidence**: 判断当前证据集是否足够做最终分类
   - **AdditionalQueryGen**: 证据不足时生成新查询
   - **Classifier**: 基于声明和完整证据集做 True/False 分类
   - 设计动机：每个 agent 只需完成一个简单子任务，降低了 LLM 的认知负担，也使得系统行为更可控可解释

2. **迭代式证据收集**:
   - 做什么：不一次检索完毕，而是动态判断证据充分性并按需继续搜索
   - 核心思路：设定 MAX_SEARCH_QUERIES（默认 4）和 MAX_SEARCH_RESULTS_PER_QUERY（默认 2）作为上限，在达到上限前持续迭代
   - 设计动机：人类验证过程天然是迭代的——看一个来源不够就继续搜索，直到有信心做出判断

3. **非自包含文档延迟处理**:
   - 做什么：当网页内容不自包含时，暂存而非立即丢弃或补充检索
   - 核心思路：随着证据集增长，之前不自包含的内容可能变得可理解；在主循环结束前再统一处理这些暂存内容
   - 设计动机：优先处理自包含证据可以减少不必要的搜索查询次数

### 损失函数 / 训练策略
- 纯推理时方法，无需训练
- 所有 agent 使用 GPT-4.1 + zero-shot prompting
- 搜索 API 使用 serper.dev

## 实验关键数据

### 主实验（GPT-4.1 作为 base LLM）

| 数据集 | 方法 | True F1 | False F1 | Macro-F1 | Weighted-F1 |
|--------|------|---------|----------|----------|-------------|
| BingCheck | FIRE | 0.89 | 0.63 | 0.76 | 0.84 |
| | **EMULATE** | **0.93** | **0.69** | **0.81** | **0.88** |
| FacTool-KBQA | FIRE | 0.89 | 0.66 | 0.78 | 0.83 |
| | **EMULATE** | **0.91** | **0.68** | **0.80** | **0.85** |
| Factcheck-Bench | FIRE | 0.87 | 0.68 | 0.78 | 0.82 |
| | **EMULATE** | **0.90** | **0.71** | **0.80** | **0.85** |

EMULATE 在所有三个数据集的 6/8 指标上取得最佳结果。

### 消融实验

| 配置 | FacTool-KBQA W-F1 | Factcheck-Bench W-F1 |
|------|-------------------|---------------------|
| EMULATE (full) | **0.85** | **0.85** |
| w/o SearchRank | 0.80 | 0.83 |
| w/o SelfContainedCheck | 0.84 | 0.82 |

### 关键发现
- **SearchRank 贡献最大（-5pp on FacTool-KBQA）**：搜索结果的质量排序对最终准确率影响关键，说明来源筛选是人类行为链中的核心步骤
- **SelfContainedCheck 在 Factcheck-Bench 上更重要（-3pp）**：该数据集声明更复杂，更多网页结果不自包含
- **弱模型也可用**：GPT-4.1-mini 上的 EMULATE 性能接近 GPT-4.1，说明框架设计降低了对单个模型能力的依赖

## 亮点与洞察
- **"模拟人类"的框架设计哲学**：按照人类验证信息的行为链拆解任务，每一步都有对应 agent，是一种可推广的多智能体任务分解策略。可迁移到其他需要判断-检索-再判断的场景（如学术综述、竞品分析）
- **SearchRank 的价值**：在 RAG 管道中加入搜索结果排序/过滤步骤是一个容易被忽略但非常有效的改进
- **迭代+延迟处理策略**：暂存不自包含的文档等证据丰富后再处理，减少了不必要的搜索调用

## 局限性 / 可改进方向
- **数据集规模小**：三个数据集均 <1000 条，且有类别不平衡（False 远少于 True）
- **仅做二分类**：现实中声明可能有 "partially true"、"misleading" 等细粒度标签
- **对搜索 API 依赖强**：搜索结果质量直接决定最终效果，在低质量搜索场景可能显著退化
- **7 个 agent 的 API 调用开销大**：每条声明需要多次 LLM 调用，成本和延迟较高
- 可改进：(a) 可以训练轻量级模型替代部分 agent（如 SearchRank 用小模型）；(b) 可以支持多跳推理的复合声明

## 相关工作与启发
- **vs FIRE (Xie et al., 2025)**: FIRE 也有迭代检索但只分 3 个组件，EMULATE 将验证流程拆解得更细（7 个 agent），特别是增加了 SearchRank 和 SelfContainedCheck
- **vs FacTool (Chern et al., 2023)**: FacTool 是单轮检索+分类的朴素管道，缺少搜索结果筛选和证据充分性判断
- **vs SAFE (Wei et al., 2024)**: SAFE 关注迭代查询生成，但未进行搜索结果的质量排序
