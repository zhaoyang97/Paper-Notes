# Play2Prompt: Zero-shot Tool Instruction Optimization for LLM Agents via Tool Play

**会议**: ACL 2025  
**arXiv**: [2503.14432](https://arxiv.org/abs/2503.14432)  
**代码**: [GitHub](https://github.com/wfangtw/play2prompt)  
**领域**: Agent  
**关键词**: tool learning, zero-shot, prompt optimization, beam search, self-reflection

## 一句话总结
提出 Play2Prompt，通过让 LLM 自主"玩"工具（试探输入输出行为）来零样本地生成工具使用示例和优化工具文档，无需任何标注数据即可显著提升 LLM agent 的工具调用能力。

## 研究背景与动机
1. **领域现状**：LLM agent 框架通过调用外部工具解决复杂任务。动态集成新工具主要依赖提示（工具文档+使用示例），工具调用质量高度依赖文档的完整性和示例的质量。
2. **现有痛点**：真实场景中用户提供的工具文档往往不完整或有噪声，且不提供使用示例。现有自动 prompt 优化方法仍需标注数据做验证，在真正的 zero-shot 场景下不可用。
3. **核心矛盾**：没有使用示例→无法优化文档→无法生成好的示例。鸡生蛋问题。
4. **本文要解决什么？** 在零标注数据条件下，自动生成高质量的工具使用示例和优化工具文档。
5. **切入角度**：模拟人类的试错过程——先"玩"工具探索其行为（正向：探索合法调用→反向：构造对应查询），再用生成的示例作为验证集优化文档。
6. **核心idea一句话**：先"玩"工具获取输入输出行为知识，再反向构造示例，最后用示例集验证优化文档——整个过程零标注。

## 方法详解

### 整体框架
Play2Prompt 分两步：**Step 1**（工具使用示例生成）：通过试错过程探索工具合法调用，反向生成查询-回答对。**Step 2**（文档优化）：用 Step 1 的示例集作为验证集，beam search 优化文档。两步都使用带自反思的 beam search 框架。

### 关键设计

1. **反向示例生成（核心创新）**：
   - 做什么：零样本生成工具使用示例 $E = (x, F_k, I_k, y)$
   - 核心思路：**反向生成**——先探索合法调用参数 $I_k$（通过 rejection sampling + self-reflection），观察工具输出，然后再构造查询 $x$ 和回答 $y$。而非传统的先想查询再调工具
   - 设计动机：文档不完整时正向调用容易失败，反向从合法调用出发保证了示例的可执行性

2. **对抗性奖励设计**：
   - 做什么：评价生成示例的质量
   - 核心思路：$R^{(i)} = R_q^{(i)} + \lambda R_e^{(i)}$，其中 $R_q$ 评价示例质量（清晰度、一致性），$R_e = -\mathcal{P}\{\mathcal{M}_T(x; \mathcal{D}_0, \varnothing); y, F_k, I_k\}$ **鼓励困难示例**（task LLM 答不对的）
   - 设计动机：类似 active learning，困难示例更能帮助 LLM 学习工具用法。示例生成（求最困难）和文档优化（求最高正确率）形成对抗目标

3. **文档优化（Step 2）**：
   - 做什么：用 beam search + self-reflection 改写工具文档
   - 核心思路：将 Step 1 生成的示例集作为验证集，每次 iteration 将 task LLM 的工具调用错误反馈给 generator LLM，引导其修正文档中的歧义和遗漏
   - 奖励：$R^{(i)} = \mathbb{E}_{(x,F,I,y) \in \mathcal{E}}[\mathcal{P}\{\mathcal{M}_T(x; \mathcal{D}^{(i)}, \varnothing); y, F, I\}]$

### 损失函数 / 训练策略
- 不涉及参数训练——纯 prompt 优化
- Beam search 在每步保留 top-reward 样本，自反思引导改进方向

## 实验关键数据

### 主实验
Berkeley Function-Calling Leaderboard (Executable)：

| 模型 | Prompting | +Play2Prompt | 提升 |
|------|----------|-------------|------|
| LLaMA-8B | 85.9% | **92.9%** | +7.0 |
| LLaMA-70B | 90.8% | 提升 | 显著 |
| GPT-4o | baseline | **提升** | 一致 |

### 消融实验
| 配置 | 结果 | 说明 |
|------|------|------|
| 仅生成示例（无文档优化）| 有提升但不如完整版 | 两步互补 |
| 仅优化文档（无示例）| 提升有限 | 示例既是 ICL 又是验证集 |
| 正向生成示例（非反向）| 大量无效调用 | 反向生成是关键 |
| 去掉困难示例偏好 | 性能下降 | 对抗性奖励有效 |

### 关键发现
- **反向生成是关键设计**：从合法调用出发构造示例，避免了文档不完整导致的调用失败
- **开放和闭源模型都受益**：LLaMA-8B/70B 和 GPT-4o 上一致提升
- **单工具示例就够了**：虽然只生成单工具使用示例，task LLM 能泛化到多工具组合任务
- **文档优化和示例生成的对抗关系**提升了整体效果

## 亮点与洞察
- **"玩工具"的隐喻非常直觉化**：模拟人类面对新工具时的探索行为——先试着用看看，观察结果，再理解用法。这种 trial-and-error 思路可以推广到任何需要理解 API 行为的场景
- **零标注的鸡蛋问题解法优雅**：通过反向生成打破"没示例就没法优化"的死循环
- **对抗性奖励**借鉴 active learning 思想：选择 task LLM 最难的示例，最大化学习效率

## 局限性 / 可改进方向
- **依赖工具可执行**：需要实际调用工具获取反馈，对不可执行或有副作用的工具不适用
- **计算成本高**：beam search × rejection sampling × self-reflection 的嵌套导致大量 LLM 调用
- **仅生成单工具示例**：虽然 task LLM 能泛化到多工具，但直接生成多工具组合示例可能更有效
- **改进方向**：(1) 减少 LLM 调用次数的高效搜索；(2) 支持无法执行的工具（通过文档推理）；(3) 多工具组合示例生成

## 相关工作与启发
- **vs ToolBench/ToolLLM**: 它们需要大量标注的工具使用数据训练，Play2Prompt 完全零样本
- **vs 自动 prompt 优化 (DSPy, RLPROMPT)**: 需要验证集，Play2Prompt 自己生成验证集
- **vs 工具文档增强**: 之前工作只优化文档或只生成示例，Play2Prompt 两者联合优化且有对抗目标
