# CoRe: Benchmarking LLMs' Code Reasoning Capabilities through Static Analysis Tasks

**会议**: NeurIPS 2025
**arXiv**: [2507.05269](https://arxiv.org/abs/2507.05269)
**代码**: [corebench.github.io](https://corebench.github.io/) (Apache-2.0)
**领域**: LLM推理
**关键词**: code reasoning, static analysis, benchmark, data dependency, control dependency, information flow

## 一句话总结

提出 CoRe，一个包含 12,553 个人工验证任务实例的高质量 benchmark，通过数据依赖、控制依赖和信息流三类静态分析基础任务，直接评估 LLM 的代码语义推理能力，揭示模型在 trace 生成和源枚举等需要多步推理的任务上仍严重不足。

## 研究背景与动机

1. **领域现状**: LLM 已广泛应用于代码生成（HumanEval）、程序修复（SWE-Bench）、漏洞检测等软件工程任务，这些任务的成功隐含地依赖对程序语义（值传播、控制流、元素间依赖关系）的深层理解。近期研究也开始将 LLM 直接用作静态分析器（如 LLMDFA、IRIS）。
2. **现有痛点**: 现有 benchmark 主要以端到端方式评估最终输出正确性（代码是否修复、生成是否通过测试），未能直接评估模型对程序语义的推理能力。动态 trace 预测类工作（如 CRUXEval）也仅关注特定输入下的运行时行为，忽略未执行分支和静态语义属性。
3. **核心矛盾**: LLM 在下游任务上表现尚可，但其底层的程序语义推理能力是否足够支撑这些任务仍然未知——缺少针对性、细粒度的评测手段来诊断模型的推理短板。
4. **本文要解决什么**: 设计一个直接评估 LLM 核心代码推理能力的 benchmark，覆盖静态分析的三大基础任务，提供比端到端评测更细粒度的诊断信息。
5. **切入角度**: 从程序分析的三大基础概念——数据依赖（def-use 链）、控制依赖（执行路径守卫）、信息流（显式+隐式传播）——出发，构建跨 C/C++、Java、Python 三语言的多任务 benchmark。
6. **核心 idea 一句话**: 与其通过端到端结果间接推断能力，不如直接考察 LLM 是否掌握了代码语义推理的"基本功"——静态分析中的依赖关系推理。

## 方法详解

### 整体框架

CoRe benchmark 构建包含四个阶段：**程序采样** → **半自动标注** → **任务设计** → **语义感知多样化采样**，最终产出 12,553 个任务实例（含 CoRe Lite 子集 1,584 个）。

### 关键设计

#### 1. 多语言程序采样
- **数据源**: 从 CodeNet（大规模竞赛代码集）和 Google Code Jam（GCJ，专家级算法解题代码）两大来源采样
- **规模**: 共 180 个程序，C/C++、Java、Python 各 60 个
- **复杂度均衡**: 按代码行数分为 4 个桶（21-40 / 41-60 / 61-80 / 81-100 LoC），每桶均匀采样
- **分析范围**: 仅关注过程内分析（intra-procedural），排除调用非库函数的函数，避免跨函数推理带来的 prompt 信息不完整问题

#### 2. 半自动标注流水线
- **初始标注**: 基于 tree-sitter 开发定制静态分析工具，自动生成数据依赖和控制依赖的初始标注
- **人工验证**: 两位有 5 年以上程序分析经验的作者独立验证，冲突由第三人裁决
- **质量保证**: 标注一致率 87.5%，共产出 6,306 个标注变量和 48,050 行标注数据
- **必要性**: 作者论证了现有程序分析工具无法跨语言自动准确提取全面的语义属性，人工介入不可或缺

#### 3. 三类核心任务
- **数据依赖 (Data Dependency)**: 判断变量 b 的值是否依赖于变量 a（通过赋值链传播），对应程序分析中的 def-use 链。应用实例：CVE-2017-5638 的污点分析（不可信 HTTP 头通过数据依赖传播到敏感操作）
- **控制依赖 (Control Dependency)**: 判断语句 ℓ₂ 的执行是否受条件语句 ℓ₁ 控制（存在分支使 ℓ₂ 必执行，另一分支使其可能不执行）。应用实例：CVE-2022-26129 的 fuzzing 路径可达性分析
- **信息流 (Information Flow)**: 判断变量值是否通过显式赋值（数据依赖）或隐式控制流（控制依赖）影响另一变量。这是最具挑战性的任务，需同时推理两种依赖。应用实例：空指针异常的程序切片与故障定位

#### 4. 两种查询类型
- **Pairwise Query**: 给定两个程序元素，判断是否存在特定依赖关系；若存在，还需生成一条从源到目标的完整 trace（传递性依赖序列）
- **Target-Centric Query**: 给定一个目标元素，枚举函数内所有对其有指定依赖关系的源元素（返回完整无序集合）

#### 5. 语义感知多样化采样 (Semantics-Aware Diverse Sampling)
- 每个程序最多采样 5 个目标，每个目标最多构建 5 个正例和 5 个负例
- **多样性保证**: 避免 trace 重叠或上下文重复，确保结构覆盖度
- **推理复杂度保证**: 选择具有非平凡依赖结构的目标
- **负样本设计**: 选择结构上"看似可能"（如在同一代码块内、语法位置相近）但实际无依赖的元素对，防止模型通过位置启发式作弊

### 评测设计
- **Prompt**: 包含详细定义、输出格式规范和 5-7 个合成小程序示例（含逐步推理解释）
- **评测指标**: 依赖分类（Precision/Recall/F1）、trace 质量（Correct Trace Rate）、源枚举（Exact Match）

| GPT o4-mini | ✓ | 86.74 | 60.43 | 32.89 |
| DeepSeek R1 | ✓ | 86.18 | 58.36 | 31.82 |
| Claude 3.7 | ✓ | 82.07 | 57.13 | 25.82 |
| Qwen3 235B | ✓ | 80.31 | 52.26 | 23.30 |
| Claude 3.5 | ✗ | 77.27 | 49.43 | 19.13 |
| DeepSeek V3 | ✗ | 75.80 | 41.08 | 13.38 |
| GPT 4o | ✗ | 74.16 | 43.56 | 14.52 |
| Llama 3.1 405B | ✗ | 68.93 | 28.48 | 3.28 |

### 分任务详细表现（Gemini 2.5 Pro / GPT o3 / DeepSeek R1）

| 任务类型 | 指标 | Gemini 2.5 Pro | GPT o3 | DeepSeek R1 |
|---------|------|---------------|--------|-------------|
| 数据依赖 | 分类 F1 | 88.53 | 93.24 | 83.29 |
| 数据依赖 | Trace CT | 90.38 | 86.23 | 67.31 |
| 数据依赖 | 枚举 EM | 49.43 | 41.89 | 38.88 |
| 控制依赖 | 分类 F1 | 92.49 | 92.11 | 92.28 |
| 控制依赖 | Trace CT | 92.26 | 77.52 | 66.62 |
| 控制依赖 | 枚举 EM | 75.66 | 70.90 | 48.37 |
| 信息流 | 分类 F1 | 94.79 | 92.13 | 83.59 |
| 信息流 | Trace CT | 68.66 | 52.37 | 39.58 |
| 信息流 | 枚举 EM | 26.73 | 15.61 | 7.12 |

### 影响因素分析

| 影响因素 | 性能变化 | 说明 |
|---------|---------|------|
| 函数长度 21-40→81-100 LoC | Trace CT 下降 8.9-15.8% | 更长的函数带来更复杂的依赖链和更多token |
| 条件语句数量增多 | Trace CT 下降 24.4-44.6% | 复杂控制流显著增加推理难度 |
| 存在 early exit (return/break) | 枚举 EM 下降最高 21.7% | 非线性控制流打断模型的顺序推理 |
| 反向依赖（源在目标之后） | 分类 F1 下降最高 49.5% | 暴露 LLM 的"从左到右"推理偏差 |
| 裁剪上下文（仅提供目标函数） | 性能提升最高 5.7% | 模型难以在冗余上下文中聚焦 |
| 要求同时分类+生成 trace | Precision 降低，Recall 升高 | 模型更倾向于预测"存在依赖"，假阳性增加 |
| Few-shot 检索增强 | 未提升甚至退化 | 标准检索器无法捕捉结构/语义相似性 |

### 关键发现

1. 推理模型一致优于非推理模型，性能差距 5.2-31.5%；Gemini 2.5 Pro 综合最优
2. 依赖判别（分类）相对容易（F1 最高 92.56%），但 trace 生成和源枚举极具挑战——最好的模型在枚举上也仅 50%
3. **信息流是最难任务**：即使分类 F1 高达 94.79%，trace 正确率仍只有 68.66%（Gemini），其他模型更低
4. 控制依赖枚举相对容易（行级别源平均仅 4.0 个），数据依赖和信息流的变量级别源分别平均 7.7 和 17.0 个，枚举难度很大
5. 反向依赖暴露了 LLM 的致命弱点：Claude 3.7 在反向依赖上 F1 下降高达 49.5%

## 亮点与洞察

- **填补评测空白**: 首个系统评估 LLM 在静态分析核心能力上的 benchmark，将评测粒度从"代码是否修好"下沉到"是否理解依赖关系"
- **人工验证的高质量数据**: 87.5% 标注一致率，半自动流水线（tree-sitter + 双人交叉验证 + 仲裁）确保标注的soundness和completeness
- **语义感知采样策略精巧**: 负样本选择结构上"看似可能"的元素，有效防止模型利用位置启发式捷径
- **反向依赖的重要发现**: 揭示 LLM 存在强烈的"从左到右"顺序推理偏差，无法构建抽象的依赖图进行双向分析。这对理解模型推理本质有重要意义
- **上下文越长反而越差**: 提供完整文件比仅提供目标函数性能更低（最高差 5.7%），说明模型难以在长上下文中聚焦相关信息——与传统分析工具天然隔离作用域的能力形成对比
- **任务间难度梯度清晰**: 分类 > Trace > 枚举，为后续改进提供了明确的难度阶梯

## 局限性 / 可改进方向

1. **仅过程内分析**: 排除跨函数调用链的推理，而真实软件的依赖经常跨越函数边界，未来需扩展到 inter-procedural
2. **程序来源单一**: CodeNet 和 GCJ 均为竞赛代码，代码风格和复杂度与真实工业项目（含类继承、设计模式、大型代码库）有显著差距
3. **缺少复杂内存模型**: 未评估指针、别名、间接内存访问等关键语义（尤其对 C/C++ 重要），这恰恰是静态分析最困难的部分
4. **信息流改进空间**: 正确率普遍低于 50%，是否需要为 LLM 设计分步推理框架（如先推数据依赖再推控制依赖最后综合）值得探索
5. **检索增强失败**: 标准 sparse/dense retriever 无法捕捉任务间的结构相似性，需要程序结构感知的检索方法
6. **规模有限**: 180 个程序、12K 任务实例在 benchmark 中算中等规模，未来可扩展到更大更多样的代码库

## 相关工作与启发

- **CRUXEval** 仅评估 Python 的输入输出行为（本质是动态分析），CoRe 定位静态分析且覆盖三种语言，评估角度互补
- **CRQBench** 从 GitHub PR 评论自动生成 C++ 代码推理问题，但自动生成容易出错且粒度偏粗，CoRe 的人工验证质量更高
- **LLMDFA / IRIS**: 将 LLM 用作静态分析器的实际应用系统——CoRe 可以作为评估这类方法基础能力的试金石
- **REval**: 评估代码覆盖率预测等运行时行为，与 CoRe 的静态分析定位不同但方法论类似
- **对 LLM-as-analyzer 范式的启示**: 模型在枚举任务上 EM 极低（最好也只有 50%），说明直接用 LLM 做完整的依赖分析仍不现实，需要与传统工具结合

## 评分

- **新颖性**: ⭐⭐⭐⭐ 首个细粒度评估 LLM 静态分析推理能力的 benchmark，填补重要空白；但 benchmark 类工作的方法创新天花板有限
- **实验充分度**: ⭐⭐⭐⭐⭐ 10 个模型（含 6 个推理模型）、3 类任务、3 种语言、多维度影响因素分析和消融实验，非常全面
- **写作质量**: ⭐⭐⭐⭐⭐ 背景从真实 CVE 案例切入动机清晰，任务定义数学化严谨，定性分析深入且有实际意义
- **价值**: ⭐⭐⭐⭐ 对理解和改进 LLM 代码推理能力有较高参考价值，反向依赖偏差等发现有启发性；但 benchmark 的实际应用距离尚远
