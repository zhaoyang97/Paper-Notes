# Born a Transformer – Always a Transformer? On the Effect of Pretraining on Architectural Abilities

**会议**: NeurIPS 2025  
**arXiv**: [2505.21785](https://arxiv.org/abs/2505.21785)  
**代码**: https://github.com/lacoco-lab/always_a_transformer  
**领域**: LLM/NLP  
**关键词**: 长度泛化, Transformer架构限制, 归纳头, 预训练偏置, 可靠性

## 一句话总结
通过系统性地研究检索和复制任务家族，揭示了大规模预训练会为Transformer引入方向性偏置（右/前向优于左/后向），但无法克服非唯一任务上的根本架构限制；微调可消除方向偏置但不能突破架构表达力边界。

## 研究背景与动机

1. **领域现状**：Transformer理论上有长度泛化限制（如无法表达某些序列任务），小模型实验已证实这些限制存在。但大规模预训练LLM是否受同样限制，一直没有定论。
2. **现有痛点**：现有研究要么只在小模型上验证理论限制（缺乏对预训练模型的实验），要么只观察LLM涌现能力而不关联理论分析。预训练可能带来全新的归纳偏置（如ICL、CoT），使理论限制失效。
3. **核心矛盾**：预训练重塑了模型参数和子电路，小模型上观察到的长度泛化失败是否直接迁移到预训练大模型？规模和数据多样性能否克服架构固有限制？
4. **本文要解决什么？** (a) 预训练后哪些架构能力被增强、哪些限制依然存在？(b) 是否存在预训练独有的偏置？(c) 微调能否消除这些偏置？
5. **切入角度**：选择检索和复制这两类基础操作作为测试床——它们是ICL、RAG等实际能力的基础构件，且有清晰的理论可证性（C-Rasp[pos]框架）。
6. **核心idea一句话**：预训练让Transformer"生来偏好"向右/前向操作（归纳电路更强），但其架构固有的长度泛化天花板始终存在。

## 方法详解

### 整体框架

研究设计了一组检索（Retrieval）和复制（Copying）任务的变体矩阵：
- **检索任务**：给定上下文序列 + 查询token，返回查询token左边或右边的token。分为唯一（Unique，查询只出现一次）和非唯一（Non-unique，查询出现多次）两类，非唯一又分首次出现和末次出现，共6个子任务（UR, UL, NRFirst, NRLast, NLFirst, NLLast）。
- **复制任务**：给定输入序列后要求模型复制，分为唯一token正向(UF)、唯一反向(UB)、非唯一正向(NF)、非唯一反向(NB) 4种。

研究包含三个递进阶段：(1) 理论分析提供长度泛化保证，(2) 预训练模型的ICL实验揭示偏置，(3) 微调+机制分析解释偏置来源。

### 关键设计

1. **C-Rasp[pos]理论框架**:
   - 做什么：为每个任务变体提供长度泛化的理论保证（或否证）
   - 核心思路：利用 Huang et al. (2025) 的框架——如果一个任务可以用 C-Rasp[pos] 程序表达，则Transformer可以长度泛化。C-Rasp[pos] 限制了位置信息和算术运算的使用
   - 理论结论：UR/UL/NRFirst/NLFirst/UF/UB 在 C-Rasp[pos] 中可表达（可泛化），NRLast/NLLast/NF/NB 不可表达（不可泛化）。关键是**左右版本没有表达力差异**（Theorem 2），预训练模型的方向性差异纯粹来自预训练偏置
   - 设计动机：建立理论基线，区分"架构不能做"和"预训练没教"

2. **归纳 vs 反归纳电路（Induction vs Anti-Induction Circuits）**:
   - 做什么：解释方向性偏置的机制来源
   - 核心思路：归纳头（Induction Head）是两层电路——第一层 previous-token head 将前一个token的嵌入写入残差流，第二层归纳头匹配当前token并复制后续token。反归纳头（Anti-Induction Head）理论上对称存在，应能向左检索。但预训练数据中归纳模式（bigram续写）远比反归纳常见，因此归纳头被选择性强化
   - 通过 patching 实验验证因果性：移除归纳头→UF准确率归零但UB不受影响；移除反归纳头→UB归零但UF不受影响

3. **自然场景验证**:
   - 做什么：将合成任务发现迁移到实际应用场景
   - **Lorem Ipsum复制实验**：让模型复制约500 token的随机Lorem Ipsum段落。通过Needleman-Wunsch算法对齐，发现所有复制错误的"转折点"几乎都发生在**歧义token**上（非唯一token，对应后续token不确定），验证了非唯一复制的理论限制
   - **Git历史操作实验**：要求模型正序（git revert）或逆序（git cherry-pick）输出commit hash。正序近乎完美，逆序显著下降——方向性偏置在实际任务中真实存在

4. **微调对偏置的消除/保留**:
   - 做什么：区分"预训练引入的偏置"和"架构固有限制"
   - 对 GPT-2 1.5B 在每个任务上做监督微调，采用位置偏移trick确保位置编码被充分训练。在OOD测试集（长度为训练集2倍）上评估
   - 结果：C-Rasp[pos]可表达的任务（UR/UL/UF/UB/NRFirst/NLFirst）微调后方向性偏置消失，OOD准确率100%；不可表达的任务（NRLast/NLLast/NF/NB）即使微调也无法完美泛化

### 训练策略
- 预训练模型评估：Llama-3.1 (8B, 70B) + Qwen2.5 (7B, 32B)，包含 completion 和 instruction-tuned 变体
- 微调：GPT-2 1.5B (APE)，仅在答案span上计算loss，训练长度 $[\ell_{min}, 100]$，测试 $[101, 200]$

## 实验关键数据

### 主实验：预训练模型 ICL 准确率

| 任务 | 方向 | Llama-3 70B | Qwen2.5-32B | 理论可泛化? |
|------|------|-------------|-------------|-------------|
| UR (唯一右) | 右 | 高 | 高 | ✓ |
| UL (唯一左) | 左 | 显著下降 | 显著下降 | ✓ |
| NRFirst | 右 | 低 | 低 | ✓ |
| NLFirst | 左 | 更低 | 更低 | ✓ |
| UF (唯一正向复制) | 前 | 近完美 | 近完美 | ✓ |
| UB (唯一反向复制) | 后 | 下降 | 下降 | ✓ |
| NF (非唯一正向) | 前 | 近完美 | 近完美 | ✗ |
| NB (非唯一反向) | 后 | 下降 | 下降 | ✗ |

核心发现：**方向性偏置**在所有模型、任务、长度上一致存在——右/前向 > 左/后向。

### 消融实验：微调后的 OOD 泛化

| 任务 | C-Rasp[pos]可表达 | 微调后方向偏置 | OOD 准确率 |
|------|-------------------|---------------|-----------|
| UR / UL | ✓ | 消失 | ~100% |
| UF / UB | ✓ | 消失 | ~100% |
| NRFirst / NLFirst | ✓ | 消失 | ~100% |
| NRLast / NLLast | ✗ | 消失 | 显著下降 |
| NF / NB | ✗ | 消失 | 显著下降 |

### 关键发现
- **方向偏置是预训练产物**：从零训练的模型不存在左右不对称，微调能消除它。归纳头在预训练中被选择性强化，反归纳头没有
- **架构限制是硬约束**：微调消除了方向偏置，但无法让不在 C-Rasp[pos] 中的任务实现完美OOD泛化。非唯一复制/检索的失败是架构固有的
- **实际影响**：Lorem Ipsum实验中，复制错误100%发生在歧义token处；Git实验中逆序准确率显著低于正序
- patching实验精确定位了因果机制：归纳头和反归纳头分别是正向和反向任务的"必要通路"

## 亮点与洞察
- **归纳-反归纳对称性分析**极为巧妙：理论上两者对称，但预训练创造了不对称——这直接解释了为什么LLM在"往回看"时表现差。这个发现对RAG（需要精确复制上下文片段）和ICL（依赖归纳头）都有实际意义
- **C-Rasp[pos]框架作为可泛化性判别器**的应用很踏实：为每个任务变体提供了 yes/no 的理论判断，实验完美匹配
- **Lorem Ipsum实验设计**是将理论限制"落地"的优秀范例：用歧义token分析精准定位了非唯一复制失败的微观机制

## 局限性 / 可改进方向
- 仅测试了 Llama-3 和 Qwen2.5 两个开源家族，未覆盖闭源模型（GPT-4o、Claude等）或循环架构（Mamba等）
- 微调实验仅在 GPT-2 1.5B 上做，未在更大模型上验证
- C-Rasp[pos]理论仅对 APE 位置编码有严格证明，对 RoPE 只有经验验证
- 未分析预训练数据中归纳模式的具体统计分布（只推测归纳比反归纳常见）
- 实际应用场景可以进一步拓展——如在代码生成、文档摘要等任务中量化方向偏置的影响

## 相关工作与启发
