# Crucible: Quantifying the Potential of Control Algorithms through LLM Agents

**会议**: NeurIPS 2025  
**arXiv**: [2510.18491](https://arxiv.org/abs/2510.18491)  
**代码**: [GitHub](https://github.com/thu-media/Crucible)  
**领域**: LLM Agent / 控制算法 / 评测  
**关键词**: control algorithm evaluation, tuning potential, LLM agent, parameter optimization, ABR, scheduling

## 一句话总结

首次提出"调优潜能"（Tuning Potential）概念并给出形式化度量，通过 LLM Agent 模拟不同能力水平的开发者对控制算法进行参数调优和逻辑级改进，在 ABR 任务上相比贝叶斯优化提升 44.1%，CartPole 上 Bang-bang 从 34→500 达到 DQN 水平。

## 研究背景与动机

1. **被忽视的"调优潜能"**：现有研究主要在默认参数或理想条件下评估算法性能，但在实际生产环境中，算法总是由领域专家针对特定场景进行调参和逻辑改造。算法的实际表现不仅取决于默认设计，更取决于其固有可调性——即"调优潜能"。
2. **超越参数灵敏度分析**：评估调优潜能不能仅停留在参数搜索层面，还需覆盖更深层的逻辑级修改（如增加控制分支、集成新组件），而这些结构性调整与开发者对算法的主观理解紧密相关。
3. **实际案例驱动**：ABR 场景中简单的 HYB 算法经 Crucible 调优后 QoE 从最差升至第一；调度场景中简单 FIFO 经调优后达到最优累积等待时间——说明调优潜能是算法选型的关键维度。

## 方法详解

### 整体架构

Crucible 包含两大核心组件：(1) LLM 驱动的多级专家模拟 Agent；(2) 基于环境特征向量的统一调优潜能度量。

### Agent 工作流

1. **领域知识注入**：通过 system prompt 注入三维度知识——任务描述（输入状态 + 输出行为空间）、优化目标（改进方向 + 评估标准）、环境概述（场景特征 + 约束条件）。
2. **工具调用**：将贝叶斯优化等工具封装为标准化函数接口，量化当前算法逻辑在超参空间中的性能提升。这对应算法潜能的"表征能力"维度——控制空间的广度和粒度。
3. **行动-反馈循环**：每次修改保存为三元组（修改理由、具体操作、观测结果），作为后续优化的经验基础，使 LLM 能从历史尝试中学习，避免重复失败。
4. **差异化能力模拟**：通过调整 Agent 的计算预算来模拟不同水平的开发者——限制贝叶斯优化调用次数（0/10/20 次）和反思迭代步数（1/2/3 步），而非设计不同 prompt。

### 调优潜能的形式化定义

1. **环境特征向量**：选取一组探测算法在环境集上运行，对每个环境得到归一化性能向量，作为环境的定量指纹。
2. **环境距离与相似度**：两个环境的距离定义为特征向量的 RMSE，相似度为 max(0, 1 - dis)。
3. **潜能公式**：算法的调优潜能为所有测试环境上相似度加权的性能增益均值，与理想环境差异大的环境增益被降权，保证度量的鲁棒性和公平性。

### 交互接口设计

控制算法向 LLM 提供两类信息：算法代码和执行日志（状态-动作-结果三元组）。LLM 仅修改控制算法代码，通过调用原始执行文件获取新测试结果。统一接口使 Crucible 可灵活支持任意类型的控制算法。

## 实验设置

- **LLM**：主要使用 Claude 3.7 Sonnet，对比 Claude 3.5 Sonnet 和 GPT-4o-mini
- **经典控制**：CartPole-v1，评估 PID、LQR、Bang-bang，以 DQN 为参考
- **ABR**：Pensieve 模拟器 + 四个公开网络数据集（Oboe/FCC/3G/Puffer），评估 BBA/MPC/HYB/BOLA/Pitree
- **调度**：Spark 模拟器 + TPC-H 查询，评估 SJF/FIFO/多级反馈/Tetris

## 实验关键结果

### 优化空间拓展
- CartPole：Bang-bang **34→500**（仅一次 LLM 逻辑修改）；PID 34→500（两轮迭代）；均达到 DQN 最优分数
- ABR：Crucible 相比贝叶斯优化在 Puffer 数据集上提升 **44.1%**
- 无贝叶斯优化辅助时，约 60% 场景 LLM 逻辑修改无显著收益；加入 20 次贝叶斯迭代后，无效场景降至 20%——说明参数级利用与逻辑级探索的协同效应至关重要

### 真实部署验证
- 使用 Dash.js 在公共 WiFi 网络上测试：调优后 HYB 和 BBA 达到 QoE **1.72**，超过 RL 基线 Pensieve 的 **1.66**
- Pitree（决策树算法）调优后无提升，印证了算法可理解性对潜能的影响

### 跨 LLM 鲁棒性
- Claude 3.7 Sonnet、Claude 3.5 Sonnet、GPT-4o-mini 三个模型的调优结论一致
- Claude 3.7 在 HYB 上表现更优（1.12 vs 1.03/1.04），更强模型可解锁更大潜能

### 潜能分析
- **表征能力影响**：HYB（带宽+缓冲双状态）的潜能（0.068）显著高于 BBA（仅缓冲，0.018），更宽状态空间 = 更大优化空间
- **可理解性影响**：Pitree 虽初始性能低但潜能也低（0.033），决策树的复杂逻辑阻碍了 LLM 理解和优化
- ABR 改进幅度远高于调度算法，因为调度 DAG 输入状态更复杂，LLM 理解难度更大

### 从潜能评估到算法优化
- **增强表征能力**：BBA 增加带宽输入 → BBA_C，初始性能仅差 0.5%，但调优后持续优于 BBA，最终提升 4%
- **简化算法逻辑**：SJF 虽初始不如多级反馈，但经 Crucible 调优后反超——简单清晰的逻辑更利于 AI 辅助优化

## 亮点与洞察

1. 首创"调优潜能"作为算法评估的新维度，不仅评估算法"有多好"，还评估"能被优化到多好"
2. 参数调优（贝叶斯）和逻辑调优（LLM）的协同设计：单靠 LLM 不稳定，单靠贝叶斯无法突破逻辑边界，两者结合才能充分挖掘潜能
3. 算法的"可理解性"和"表征能力"是影响潜能的两个核心因素——设计简单可理解、状态空间丰富的算法
4. 通过资源预算（而非 prompt）模拟不同能力开发者的思路值得借鉴

## 局限性

1. **稳定性问题**：不同 LLM 版本和能力会影响结果，但作者认为这恰好模拟了不同水平开发者
2. **黑盒算法受限**：目前无法直接修改黑盒算法内部逻辑，仅讨论了由黑盒蒸馏出的决策树
3. **LLM 逻辑修改成功率低**：无贝叶斯辅助时 60% 场景无改进，依赖贝叶斯优化兜底
