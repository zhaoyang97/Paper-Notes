# Text-to-Code Generation for Modular Building Layouts in Building Information Modeling

**会议**: NeurIPS 2025  
**arXiv**: [2509.23713](https://arxiv.org/abs/2509.23713)  
**代码**: [GitHub](https://github.com/CI3LAB/Text2MBL)  
**领域**: NLP理解 / 代码生成 / BIM自动化  
**关键词**: Text-to-Code, BIM, 模块化建筑布局, 代码生成, LLM微调

## 一句话总结
提出 Text2MBL 框架，将自然语言描述转化为可执行的 BIM 代码（而非坐标序列），通过面向对象的代码架构和 LLM 微调实现模块化建筑布局的自动生成，在几何一致性上比坐标驱动方法提升 10%+ IoU。

## 研究背景与动机

1. **领域现状**：模块化建筑（Modular Construction）将标准化3D体积单元在工厂预制后现场组装，是建筑业发展趋势。BIM（Building Information Modeling）是建筑全生命周期的核心数字化工具，编码了语义丰富的结构信息。近年来文本驱动设计成为新兴方向。

2. **现有痛点**：
   - 现有文本到布局方法（如 Tell2Design、Text2BIM）输出坐标序列或像素图，但坐标方法在组件数量增多时性能急剧下降，因为 LLM 需要做复杂的空间几何推理
   - 图片输出需要后处理（光栅转矢量）才能转为 BIM 格式，容易出现空间冲突和语义不一致
   - 模块化建筑特有的三层层次结构（Module → Unit → Room）比传统布局复杂得多，现有方法无法处理
   - 从概念设计到施工工作流之间存在输出格式兼容性鸿沟

3. **核心矛盾**：LLM 擅长序列生成但不擅长空间几何推理；模块化建筑需要精确的层次约束和几何一致性——需要一个表示方式能将空间推理简化为序列理解。

4. **本文要解决什么**：(1) 设计一种代码驱动的 MBL 表示方式，让 LLM 只需生成高层语义动作序列而非底层坐标；(2) 构建首个文本-BIM代码配对数据集；(3) 验证代码范式显著优于坐标范式。

5. **切入角度**：利用 BIM 天然的面向对象特性，将布局生成抽象为一系列代码操作（创建模块、分割、合并、分配房间等），每个操作封装了底层几何逻辑。这样 LLM 只需理解语义而非推导坐标。

6. **核心idea一句话**：用面向对象代码作为中间表示，将文本到 BIM 布局问题重新表述为 sequence-to-sequence 代码生成任务。

## 方法详解

### 整体框架
输入是自然语言的建筑描述（如"两个模块，北侧模块包含一室一厅..."），输出是可在 Autodesk Revit 中直接执行的 C# 代码序列。代码按时间步执行：先创建模块 → 定义单元 → 分配房间 → 放置门窗等建筑元素。

### 关键设计

1. **面向对象代码架构（Module/Unit/Room/Utils）**：
   - 做什么：定义四个核心类，封装模块化建筑的层次结构和操作
   - 核心思路：`Module` 支持绝对坐标和相对位置创建、分割和合并；`Unit` 由一组 Module 组合而成；`Room` 可基于模块/单元、方向、角落、相对位置等多种方式分配；`Utils` 提供门窗创建、分割合并等静态方法
   - 设计动机：将复杂的几何推理（碰撞检测、凹多边形处理等）封装在底层，LLM 只需调用高层接口。例如"在模块1南侧创建厨房"只需一行 `Room kitchen = new Room(name: "Kitchen", module: m1, direction: "south", dimension: 1800)`
   - 关键优势：**相对定位**替代绝对坐标——新模块相对于已有模块定义位置，避免了 LLM 进行绝对坐标计算

2. **概念形式化**：
   - 做什么：将 MBL 定义为六元组 $L = \langle \mathcal{M}, \mathcal{U}, \mathcal{R}, \mathcal{E}, \mathcal{A}, \mathcal{C} \rangle$
   - 核心思路：模块、单元、房间三层层次必须满足嵌套约束：每个单元完全包含在模块的并集中，每个房间嵌套在某个单元内。除了邻接和连通关系，还引入 **conjoint** 关系表示同一模块内的房间共存
   - 设计动机：为代码架构提供数学基础，确保生成的布局满足模块化建筑的物理约束

3. **数据收集与合成数据增强**：
   - 做什么：从香港建筑署收集 198 个真实模块化建筑设计，每个标注可执行 BIM 代码 + 两种文本描述
   - 核心思路：采用三种数据增强策略：(a) 部分合成——用模板或 GPT 从已有代码反向生成描述；(b) 全合成——按语法规则生成新代码再生成描述。混合原始数据训练比单独用合成数据效果好
   - 两种代码风格：named arguments（显式参数名）和 positional arguments（按位置传参），构成 $\mathcal{D} = \{(d_i^a, d_i^b, c_i^{\text{name}}, c_i^{\text{pos}})\}$
   - 设计动机：MBL 领域数据极度匮乏（仅 198 个样本），合成增强是必要手段

4. **LLM微调**：
   - 做什么：微调 Qwen2.5 系列模型（0.5B-7B，含 vanilla/Coder/Math 变体）
   - 核心思路：将任务建模为条件代码生成 $\hat{c} = \arg\max_c p_\theta(c | d)$，用 LoRA 微调
   - 评估三维度：**可执行有效性**（compile rate + pass rate）、**语义保真度**（instance F1 + argument F1）、**几何一致性**（IoU）

## 实验关键数据

### 主实验：代码 vs 坐标

| 模型 | 输出格式 | Overall IoU | Module IoU | Unit IoU | Room IoU |
|------|---------|-------------|------------|----------|----------|
| Qwen2.5-1.5B | 坐标 | 78.13% | 86.98% | 80.27% | 69.85% |
| Qwen2.5-1.5B | 代码 | 91.65% | 95.47% | 93.79% | 86.79% |
| Qwen2.5-7B | 坐标 | 85.38% | 90.35% | 90.39% | 77.33% |
| Qwen2.5-7B | 代码 | 95.83% | 98.51% | 98.11% | 91.64% |
| Qwen2.5-Coder-7B | 坐标 | 84.64% | 90.12% | 88.18% | 77.53% |
| Qwen2.5-Coder-7B | 代码 | **98.43%** | **98.78%** | **99.19%** | **97.37%** |

### 消融实验

| 配置 | 关键发现 |
|------|---------|
| 组件数量增多 | 坐标方法 IoU 下降 sharp，代码方法更鲁棒 |
| 仅 10% 原始数据 + 合成增强 | 性能大幅提升，证明合成数据有效 |
| Named vs Positional args | Named 在抽象指令下编译率更高（更鲁棒） |
| Coder vs vanilla vs Math | Coder 系列整体最优，Math 次之 |

### 关键发现
- **代码比坐标全面碾压**：代码驱动在所有模型尺寸上的 IoU 均显著超过坐标驱动（+10-15%），且随组件数量增多优势更大
- **相对定位是关键**：代码架构通过相对位置描述消除了绝对坐标计算的需求，这直接解释了为什么 LLM 用代码比用坐标更擅长
- **语义保真度高于通过率**：模型更擅长理解和提取参数信息（argument F1 高），但完美可执行代码仍有挑战（pass rate 较低），提示可用 argument 提取做后处理修复
- **合成数据增强有效**：在极低数据场景（10% 原始数据）下，模板合成和模型合成增强均显著提升性能
- **Qwen2.5-Coder-7B 最优**：达到 98.43% IoU，接近完美

## 亮点与洞察
- **代码作为中间表示的范式非常优雅**：将空间推理问题转化为语义代码生成，利用 LLM 的代码能力而非几何推理能力，思路可推广到 CAD、电路设计等领域
- **相对定位消除绝对坐标的设计**：模块间用 "south"/"east" + offset 而非 (x,y) 坐标，大幅降低了 LLM 的推理难度
- **概念形式化严谨**：将模块化建筑的层次结构形式化为数学定义，使代码架构有清晰的理论支撑

## 局限性 / 可改进方向
- **数据规模极小**：仅 198 个真实设计，泛化性有待验证
- **仅限参数化设计**：当前框架要求用户给出精确的空间/几何参数，不支持模糊或高层概念性描述（如"宽敞的客厅"）
- **建筑元素种类单一**：使用相同的墙、地板、门族，未考虑材质、宽度等变化
- **未做用户研究**：缺乏对设计师实际使用体验的评估
- **pass rate 仍有提升空间**：虽然 IoU 很高，但代码编译通过率在某些设置下仍需后处理修复

## 相关工作与启发
- **vs Text2BIM/Tell2Design**：它们输出坐标序列，需要 LLM 做空间推理；Text2MBL 输出代码，将空间推理封装在代码层，效果差距达 10%+ IoU
- **vs Text2CAD**：类似思路用于 CAD 建模，但未处理建筑领域特有的层次约束和语义丰富的 BIM 格式
- **代码作为中间表示的趋势**：与 AutomaTikZ（科学矢量图）、DiagrammerGPT（开放域图表）等工作一脉相承，代码/符号文法作为可解释中间表示是设计自动化的共同方向

## 评分
- 新颖性: ⭐⭐⭐⭐ 代码驱动范式替代坐标在建筑领域是首次，BIM 代码架构设计精巧
- 实验充分度: ⭐⭐⭐⭐ 多模型多尺寸对比充分，但数据规模小限制了结论外推
