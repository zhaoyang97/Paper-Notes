# Classical Planning with LLM-Generated Heuristics: Challenging the State of the Art with Python Code

**会议**: NeurIPS 2025  
**arXiv**: [2503.18809](https://arxiv.org/abs/2503.18809)  
**代码**: 公开可用  
**领域**: AIGC检测 / AI规划  
**关键词**: 经典规划, LLM代码生成, 启发式函数, PDDL, 搜索算法

## 一句话总结
让 LLM 为经典规划问题生成 Python 启发式函数代码，从 n 个候选中选最优，在 IPC 2023 基准上用纯 Python 规划器超越了 C++ 实现的 SOTA 启发式（如 hFF），且保证所有计划正确。

## 研究背景与动机

1. **领域现状**：经典规划（如 PDDL）依赖启发式搜索。启发式函数要么是领域无关的（通用但不精确），要么是人工为特定领域手工设计的（需要大量专家经验），要么是逐领域学习的（每个域需重训）。

2. **现有痛点**：
   - LLM 直接生成计划不可靠，经常产生不正确的计划
   - 专门的 LLM "推理" 模式计算成本高且仍不如专用规划器
   - 领域无关启发式在很多领域上不够精确

3. **核心矛盾**：LLM 有领域理解能力但不能可靠地规划；传统规划器有可靠的搜索但缺少领域特定知识。

4. **切入角度**：让 LLM 生成启发式函数代码（而非直接生成计划），再由传统搜索算法使用这些启发式——保证正确性由搜索算法保证，启发式质量由 LLM 的领域理解提供。

5. **核心idea一句话**：LLM 不直接规划，而是生成域相关启发式函数的 Python 代码，注入到规划器中使用。

## 方法详解

### 整体框架
给 LLM 提供 PDDL 域描述 + 示例 + Pyperplan 代码接口 → LLM 生成 n 个候选启发式函数（Python） → 在训练任务上用 GBFS 评估 → 选最优启发式 → 在测试任务上使用。整个流程**无需 LLM-规划器间的迭代交互**。

### 关键设计

1. **Prompt 设计**:
   - 做什么：为 LLM 提供充分的上下文以生成高质量启发式
   - 核心思路：Prompt 包含7项内容——PDDL 域文件、最小/最大训练任务、两个示例域（Gripper/Logistics）的域文件+任务+启发式代码、Pyperplan 状态表示示例、规划任务/动作 Python 代码、常见陷阱清单
   - 设计动机：示例启发式展示了代码风格和接口使用，checklist 帮助避免常见错误

2. **采样+选择策略**:
   - 做什么：生成 n 个候选启发式，评估后选最优
   - 核心思路：温度设为 1.0 增加多样性，每个候选在训练集上运行 GBFS（5分钟限制），按解题数排序，平局用 agile score 打破
   - 设计动机：LLM 输出有随机性，多次采样能产生质量差异大的候选；选择过程成本固定（每域只做一次）

3. **正确性保证**:
   - 做什么：确保所有找到的计划都正确
   - 核心思路：启发式只影响搜索效率，不影响正确性——GBFS 只产生正确的计划
   - 设计动机：这是相对于端到端 LLM 规划的根本优势

## 实验关键数据

### 主实验（IPC 2023 Learning Track，8 个域，每域 90 个测试任务）

| 方法 | 解题总数 | 实现 |
|------|---------|------|
| 端到端 LLM 规划 | 很少 | - |
| hFF (C++ Fast Downward) | 基线 | C++ |
| SOTA 学习 planner | 竞争力强 | C++ |
| LLM 生成启发式 (Pyperplan) | **最多** | Python |

### 关键发现
- LLM 生成的启发式在**纯 Python 实现**下仍超越 C++ 实现的 hFF 和最强学习规划器
- 非推理 LLM 生成的启发式远优于端到端 LLM 规划
- 在某些域中，LLM 启发式的**状态扩展数更少**，说明启发式本身更具信息量（而非仅靠暴力搜索）
- 测试任务远大于训练任务（如 Blocksworld: 训练 n∈[2,29], 测试 n∈[5,488]），展示了强泛化能力

## 亮点与洞察
- **代码生成而非计划生成**：巧妙地避开了 LLM 不能可靠规划的痛点，让 LLM 做它擅长的事（写代码），把可靠性交给搜索算法。这个"生成工具而非直接解题"的范式非常优雅
- **Python 超越 C++**：纯 Python 实现能超越高度优化的 C++ 规划器，说明启发式质量的重要性远超实现效率
- **每域固定成本**：生成 n 个候选只需 n 次 LLM 调用，选定后可用于无限新任务，成本模型极好

## 局限性 / 可改进方向
- Pyperplan 不支持部分 IPC 域（Ferry, Satellite），限制了评估覆盖
- 启发式选择仅基于训练集，可能在分布外测试中次优
- 未探索迭代改进（如根据失败案例让 LLM 修改启发式）
- LLM 生成的启发式可解释性未深入分析

## 相关工作与启发
- **vs 端到端 LLM 规划**：LLM 直接规划不可靠且不泛化，本文方法保证正确性且泛化到大任务
- **vs 学习型规划器**：如 LAMA-first，需要 per-domain 训练；本文方法用 LLM 零样本生成
- **启发**："让 AI 生成工具/代码而非直接解题"是一个强大的通用范式

## 评分
- 新颖性: ⭐⭐⭐⭐⭐ 问题定义精彩——将 LLM 从规划器变为启发式生成器
- 实验充分度: ⭐⭐⭐⭐⭐ IPC 2023 标准基准，8 个域，与多个强基线对比
- 写作质量: ⭐⭐⭐⭐⭐ 动机清晰，方法简洁
- 价值: ⭐⭐⭐⭐⭐ 开创性工作，对 AI 规划和 LLM 应用都有深远影响
