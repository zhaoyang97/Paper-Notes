# SQL-R1: Training Natural Language to SQL Reasoning Model By Reinforcement Learning

**会议**: NeurIPS 2025  
**arXiv**: [2504.08600](https://arxiv.org/abs/2504.08600)  
**代码**: https://github.com/IDEA-FinAI/SQL-R1  
**领域**: LLM推理  
**关键词**: NL2SQL, 强化学习, GRPO, 冷启动, 合成数据, 推理能力

## 一句话总结
首次系统地将 GRPO 强化学习应用于 NL2SQL 任务，通过四层递进式奖励函数和 200K 冷启动 + 5K 复杂样本 RL 训练策略，7B 模型在 Spider 和 BIRD 上分别达到 88.7% 和 66.6%，超越 GPT-4 同规模模型。

## 研究背景与动机

1. **领域现状**：NL2SQL 使非专业用户可直接用自然语言查询数据库，当前主要依赖 SFT 方法，闭源模型（GPT-4）成本高且不可控。
2. **现有痛点**：SFT 方法在复杂数据库场景（多表关联、嵌套查询）中表现受限，固定的生成策略难以处理歧义语义，泛化能力差。
3. **核心矛盾**：SFT 的"数据规模瓶颈"——需要大量高质量标注数据但标注成本高，且增加数据量不一定提升复杂场景的推理能力。
4. **本文要解决什么**：① RL 能否替代 SFT 用于 NL2SQL？② 如何设计 NL2SQL 特定的奖励？③ 最少需要多少数据？
5. **切入角度**：受 R1 系列模型启发，将 GRPO 算法迁移到 SQL 生成任务，设计包含执行验证的奖励信号。
6. **核心 idea 一句话**：冷启动 SFT 激活基础能力 + 5K 复杂样本 GRPO 训练 + 四层递进奖励 = 小数据大性能。

## 方法详解

### 整体框架
两阶段训练：① 冷启动——200K 合成数据 SFT 激活 NL2SQL 基础能力；② RL 阶段——5K 复杂样本 + GRPO 算法优化推理策略。推理时生成 8 个 SQL 候选，通过自洽投票选最优。

### 关键设计

1. **四层递进式奖励函数**:
   - 做什么：格式 → 执行 → 结果 → 长度，逐层评估 SQL 质量
   - 核心思路：格式奖励 $S_f$（±1）检查 think/answer 标签；执行奖励 $S_e$（±2）验证 SQL 可执行性；结果奖励 $S_r$（±3）判断查询结果正确性（最高优先级）；长度奖励 $S_l$ 鼓励详细推理过程
   - 设计动机：严格优先级确保模型不会为了长度牺牲正确性，循序渐进的学习信号让模型从格式→语法→语义逐步掌握

2. **GRPO 强化学习算法**:
   - 做什么：每个查询生成 G=8 个 SQL 候选，基于组内相对性能计算优势
   - 核心思路：无需价值模型，通过重要性采样比率 $r_i^{ratio}$、剪裁和 KL 散度正则化控制策略更新。相比 PPO 更轻量
   - 设计动机：NL2SQL 的离散奖励（执行成功/失败）天然适合 GRPO 的组内比较机制

3. **冷启动 + 5K 复杂数据策略**:
   - 做什么：200K 合成 SFT 数据激活基础能力，5K 复杂样本进行 RL 训练
   - 核心思路：冷启动使用 CoT 标注的合成数据（SynSQL-200K），RL 阶段故意不提供 CoT，强制模型自主探索推理策略
   - 设计动机：证明 RL 的数据效率——仅 5K 复杂样本就能显著超越使用 200K 全量数据的 SFT

### 损失函数 / 训练策略
冷启动用标准 SFT 损失，RL 阶段用 GRPO 目标函数。基础模型 Qwen2.5-Coder-7B，推理时 8 路自洽投票。

## 实验关键数据

### 主实验（Spider & BIRD）

| 模型 | 参数 | Spider-Test | BIRD-Dev |
|------|------|-----------|---------|
| CodeS (SFT) | 15B | 85.4% | 59.4% |
| GPT-4 (DAIL-SQL) | - | 86.6% | 57.4% |
| OmniSQL (SFT) | 7B | 88.9% | 66.1% |
| **SQL-R1** | **7B** | **88.7%** | **66.6%** |
| SQL-R1 | 14B | 88.1% | **67.1%** |

7B 模型超越大多数 GPT-4 方案，与最强 SFT baseline 持平/超越。

### 消融实验（BIRD 按难度分层）

| 难度 | SQL-R1 (7B) | SQL-R1 (14B) | GPT-4 |
|------|-----------|------------|-------|
| Simple | 72.1% | 72.4% | 66.9% |
| Moderate | 60.8% | 59.7% | 46.5% |
| **Challenging** | 51.0% | **56.5%** (+5.5%) | 43.8% |

### 关键发现
- **RL 优于 SFT**：在 BIRD 上 SQL-R1 (RL) 7B > OmniSQL (SFT) 7B，使用相同基础模型证明 RL 的优势
- **5K 数据的惊人效率**：仅 5K 复杂样本 RL 训练即可达到 SOTA，大幅降低数据需求
- **扩展时针对性提升**：7B→14B 扩展主要提升困难级别（+5.5%），简单/中等级别提升微小，说明更多参数帮助处理长程依赖
- **成本效益突出**：7B 开源模型比 GPT-4 省费用 70%+，性能不低
- **推理可解释性**：输出 `<think>` 过程，满足高风险领域合规需求

## 亮点与洞察
- **首次 RL for NL2SQL**：系统验证了 GRPO 在结构化输出生成中的有效性，开辟新方向
- **四层奖励的精妙设计**：格式→执行→结果→长度的递进与 NL2SQL 任务特性完美匹配——SQL 必须先可执行再谈正确
- **冷启动的关键作用**：200K SFT 提供稳定基础，后续 RL 仅需 5K 精选数据，证明 SFT+RL 两阶段范式的高效性
- **方法可迁移**：RL+奖励设计框架可推广到其他结构化生成任务（JSON/XML/正则等）

## 局限性 / 可改进方向
- BIRD 67.1% vs GPT-4o 73%，复杂真实场景仍有差距
- 合成数据可能无法覆盖真实 DBA 查询的边界情况
- RL 训练计算成本未披露（8 路候选×执行评估）
- 执行准确率（EX）不能处理语义等价但 SQL 写法不同的情况
- 缺少冷启动策略的消融（无 SFT vs 有 SFT、不同数据量曲线）
- 未在金融/医疗等真实跨域场景验证

## 相关工作与启发
- **vs OmniSQL**: 同基础模型 SFT 方案，SQL-R1 用 RL 在 BIRD 上超越 0.5%，证明 RL 的额外收益
- **vs SQL-o1**: 通过长思考链提升推理，SQL-R1 通过奖励信号提升，两种思路互补
- **vs MCTS-SQL**: 树搜索 vs 策略学习，SQL-R1 无需搜索开销，推理更高效

## 评分
- 新颖性: ⭐⭐⭐⭐ 首次将 GRPO 系统应用于 NL2SQL，四层奖励设计新颖，但 GRPO 算法本身非原创
- 实验充分度: ⭐⭐⭐⭐ Spider+BIRD+衍生集+难度分层分析全面，但缺冷启动消融和训练成本分析
- 写作质量: ⭐⭐⭐⭐ 结构清晰，问题设置明确，但部分细节（自洽投票机制）描述不足
- 价值: ⭐⭐⭐⭐⭐ 开源、低成本、高性能的 NL2SQL 方案，对企业部署有直接价值
