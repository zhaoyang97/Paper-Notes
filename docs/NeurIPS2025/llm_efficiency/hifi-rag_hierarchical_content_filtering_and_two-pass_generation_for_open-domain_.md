# HiFi-RAG: Hierarchical Content Filtering and Two-Pass Generation for Open-Domain RAG

**会议**: NeurIPS 2025  
**arXiv**: [2512.22442](https://arxiv.org/abs/2512.22442)  
**代码**: 无  
**领域**: LLM Agent / RAG  
**关键词**: 检索增强生成, 层级过滤, 两阶段生成, 多LLM策略, Flash-Pro级联

## 一句话总结
通过分离轻量级 Flash 模型的过滤能力与 Pro 模型的推理能力，构建多阶段管道（查询优化→分层过滤→两阶段生成→引文验证），在 MMU-RAGent 竞赛中实现 SOTA 性能。

## 研究背景与动机

1. **领域现状**：RAG 在开放域环境中面临信息检索挑战，标准向量相似度搜索容易检索无关信息。

2. **现有痛点**：
   - 检索到的文档中包含大量无关信息（垃圾进垃圾出）
   - 生成的答案与用户意图不对齐
   - 处理后知识截断数据时性能下降

3. **核心矛盾**：需要同时保证检索精度、生成质量和计算成本的均衡。

4. **本文要解决什么？** 在开放域 RAG 中实现高质量的内容检索和答案生成。

5. **切入角度**：采用级联模型策略，用轻量级 Flash 模型进行语义过滤，用高性能 Pro 模型进行推理。

6. **核心idea一句话**：多阶段管道系统，将查询优化、分层过滤、两轮生成和引文验证逐步实现。

## 方法详解

### 整体框架
HiFi-RAG 包含五个阶段：(1) 查询规划——Flash 优化用户查询；(2) 检索与 URL 过滤——Google 搜索 + Flash 级联过滤；(3) 分层内容解析与过滤——HTML 解析 + LLM 重排序；(4) 两阶段生成——Draft + Refinement（均用 Pro）；(5) 引文验证——Flash 验证源归属。

### 关键设计

1. **分层内容过滤**:
   - 做什么：将 HTML 按标题（h1-h4）分组为层级化分块，Flash 基于查询评估每个分块相关性
   - 核心思路：仅用标题+前 200 字符做判断，平均移除 60.5% 的分块
   - 设计动机：相比全文向量相似度，基于 LLM 的分块过滤更懂语义

2. **两阶段生成**:
   - 做什么：分离事实性和风格——Pro 先生成全面答案（Draft），再调整风格和长度（Refinement）
   - 设计动机：单次生成难以同时优化事实性和呈现风格

3. **Flash-Pro 级联模型**:
   - 做什么：用 Flash 处理高吞吐量任务（成本低 4-6x），用 Pro 进行推理密集任务
   - 设计动机：充分利用模型能力梯度，成本和性能的理想平衡

## 实验关键数据

### 主实验

| 系统配置 | ROUGE-L (F1) | DeBERTaScore (F1) | 改进 |
|---------|-------------|-------------------|------|
| Baseline (原始查询) | 0.2291 | 0.6375 | - |
| +查询改写 | 0.2591 | 0.6667 | +13% |
| +RAG 搜索 | 0.2664 | 0.6677 | +16.3% |
| +过滤 | 0.2695 | 0.6712 | +17.7% |
| 最终完整系统 | **0.2739** | **0.6772** | +19.6% |

### Post-cutoff 数据分析

| 配置 | Test2025 ROUGE-L | Test2025 DeBERTaScore |
|------|-----------------|----------------------|
| 基线无搜索 | 0.2766 | 0.6574 |
| RAG+过滤 | 0.3031 | 0.6840 |
| 完整系统 | **0.3182** | **0.7092** |

### 关键发现
- Post-cutoff 数据上 ROUGE-L 提升 57.4%，说明 RAG 对解决 LLM 知识过时问题价值巨大
- URL 过滤减少 33.5% 的 URL 数量，显著降低延迟
- 嵌入式过滤性能不如 LLM（可能因拓扑噪声）
- DSPy 优化易过拟合到验证集

## 亮点与洞察
- **Flash-Pro 级联架构的优雅性**：充分利用模型能力梯度，成本效益优
- **分层过滤有效性**：60.5% 分块移除率说明大多数检索内容是噪声
- **两阶段分离事实性和风格**：简单而有效的设计决策

## 局限性 / 可改进方向
- Flash 虽便宜但仍需多次调用，系统延迟可能在某些场景不可接受
- 两阶段生成依赖手选示例，泛化性受限
- URL 过滤可能误杀有价值信息

## 相关工作与启发
- **vs 传统 RAG**：传统向量检索语义理解有限，HiFi-RAG 用 LLM 做过滤更精准
- **vs 完整 Agent 方案**：Agent 方案成本 10x 高且无性能收益
