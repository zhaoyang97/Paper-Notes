# From Judgment to Interference: Early Stopping LLM Harmful Outputs via Streaming Content Monitoring

**会议**: NeurIPS 2025
**arXiv**: [2506.09996](https://arxiv.org/abs/2506.09996)
**代码**: [Project](https://liesy.github.io/SCM)
**领域**: AI安全 / 内容审核
**关键词**: streaming moderation, harmful content detection, early stopping, token-level annotation, partial detection

## 一句话总结
提出 Streaming Content Monitor (SCM)——首个原生支持部分检测的流式有害内容监控器，通过 FineHarm 数据集（29K 样本含 token 级标注）和层次一致性感知学习，平均仅需看到 18% 的 response tokens 即可达到 0.95+ macro F1，实现对 LLM 有害输出的实时早停。

## 研究背景与动机

1. **领域现状**：LLM 服务商通常在安全对齐之外部署内容审核器作为外部安全护栏（如 OpenAI Moderation API、LlamaGuard），但现有审核器采用"完整检测"——必须等 LLM 输出完整后才判断有害性。
2. **现有痛点**：(a) 完整检测导致高延迟——有害内容已全部生成/展示给用户才被拦截；(b) 现有部分检测方案（如 ProtectAI、GuardrailsAI）直接将完整检测模型应用于不完整输出，存在训练-推理 gap；(c) 这些方案需要在每个 token 步骤重复编码全部已生成内容，冗余成本高。
3. **核心矛盾**：越早检测越好（用户体验），但信息越不完整判断越难（检测准确率）。
4. **切入角度**：设计原生支持流式检测的数据和模型——FineHarm 提供 token 级监督，SCM 在训练阶段就学习基于不完整语义做判断。

## 方法详解

### 整体框架
(1) FineHarm 数据集：29K 样本，token 级有害性标注；(2) SCM 模型：特征提取器 + token 评分器（推理时用）+ 整体评分器（训练时用，推理时丢弃）；(3) 层次一致性感知学习：用 response 级知识指导 token 级判断。

### 关键设计

1. **FineHarm 数据集构建**:
   - 收集 43K+ 标注 response（来自 WildGuard 9 个 LLM + 无审查 LLM 生成）
   - Token 级标注：启发式方法——用 POS 标签过滤功能词，仅标注有害句中的实义词（名词/动词/形容词）为有害
   - 关键统计：有害 response 中仅 46% 句子有害、32.8% 词有害——确认细粒度标注的必要性
   - 有害词在 response 中均匀分布——模型无法靠位置捷径

2. **SCM 训练策略**:
   - **双级监督**：同时训练 token 级评分器（每个 token 的有害概率 $c_{t_i}$）和 response 级整体评分器（$c_{hol}$）
   - **逻辑一致性约束**：$F = f_{hol}(h_n) \Rightarrow g(\{f_{tok}(h_i)\})$——如果 response 有害，则至少有一个 token 被判有害；如果 response 无害，所有 token 应无害
   - 损失：$L = \alpha L_{tok} + (1-\alpha) L_{hol} + \beta L_{logic}$
   - 训练时，整体评分器将全局知识注入 token 表征；推理时丢弃整体评分器，只用 token 评分器做流式判断

3. **Delay-k 部分检测**:
   - 不在单个 token 就触发，而是累积 k 个有害 token 后才判定 response 有害并终止
   - 公式：$\text{Harm}_{@k} = 1$ 当且仅当 $\sum_{j=1}^{i} \mathbb{1}(c_{t_j}) \geq k$
   - k 越大越稳健（减少误报），但停止越晚（更多有害内容泄露）

## 实验关键数据

### 部分检测性能

| 方法 | Macro F1 | 平均检测位置 | 说明 |
|------|---------|------------|------|
| LlamaGuard (full) | 0.958 | 100% | 完整检测基线 |
| OpenAI Mod (full) | 0.889 | 100% | 完整检测基线 |
| LlamaGuard (partial, k=3) | 0.776 | ~40% | 有训练-推理 gap |
| **SCM (k=3)** | **0.955** | **18%** | 原生部分检测 |

### 与完整检测对比

| SCM 变体 | F1 (harmful) | F1 (benign) | Macro F1 |
|---------|-------------|-------------|---------|
| SCM-full (完整) | 0.972 | 0.968 | 0.970 |
| SCM-partial (k=3) | 0.962 | 0.949 | 0.955 |

### 作为安全对齐工具

| 方法 | 无害分数 | 有用分数 | 说明 |
|------|---------|---------|------|
| DPO | 73.2 | 67.5 | 标准对齐 |
| **DPO + SCM 伪标注** | **78.9** | **68.1** | SCM 提供 token 级反馈 |

### 关键发现
- **18% tokens 即达 0.95+ F1**：SCM 平均只需看到不到 1/5 的 response 就能准确判断
- **与完整检测几乎无差距**：部分检测 F1 仅比完整检测低 0.015
- **超越直接应用完整检测模型**：LlamaGuard 做部分检测时 F1 降至 0.776，SCM 保持 0.955
- **可辅助安全对齐**：作为伪标注器用于 DPO，无害分数提升 5.7 个点

## 亮点与洞察
- **原生流式设计**：不是将完整检测模型硬套到部分输出上，而是从数据标注到模型训练都为"不完整语义"优化
- **逻辑一致性约束优雅**：用命题逻辑蕴含关系连接 token 级和 response 级预测，比多任务硬损失更自然
- **FineHarm 标注方法实用**：POS 启发式方法虽简单但经实验验证比 LLM 标注和 likelihood 差异法更可靠
- **双重身份**：SCM 既是推理时安全护栏，又能作为训练时的伪标注器改善对齐

## 局限性 / 可改进方向
- **POS 启发式标注有噪声**：并非所有有害句中的实义词都有害，可能引入假阳性
- **模型大小限制**：SCM 基于小模型（编码器），对复杂/隐式有害内容的理解有限
- **仅英文评估**：多语言场景未测试
- **改进方向**：(1) 用 RL 或主动学习优化 token 级标注；(2) 结合 prompt 意图分析提升早期检测；(3) 适配多模态内容审核

## 相关工作与启发
- **vs LlamaGuard**：LlamaGuard 为完整检测设计，直接用于部分检测 F1 降 18 点；SCM 原生支持部分检测
- **vs Constitutional Classifier (Sharma 2025)**：基于宪法规则训练分类器，但仍需完整输出；SCM 支持流式
- **启发**：在 LLM 服务中，安全审核的延迟直接影响用户体验和安全性——"快一步检测"比"准一点检测"可能更重要
