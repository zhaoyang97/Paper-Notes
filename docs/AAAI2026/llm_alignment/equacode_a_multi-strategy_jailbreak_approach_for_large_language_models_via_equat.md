# EquaCode: A Multi-Strategy Jailbreak Approach for Large Language Models via Equation Solving and Code Completion

**会议**: AAAI 2026  
**arXiv**: [2512.23173](https://arxiv.org/abs/2512.23173)  
**代码**: [GitHub](https://github.com/lzzzr123/Equacode)  
**领域**: 对齐RLHF / AI安全  
**关键词**: 越狱攻击, LLM安全, 方程求解, 代码补全, 跨域攻击

## 一句话总结
提出EquaCode多策略越狱方法，将恶意查询分解为方程求解（B+C+x=A）和代码补全（补全Solver类的solve()方法）的跨域组合，在GPT系列上平均攻击成功率92.78%，在最新模型（Gemini/DeepSeek/Grok）上接近100%。

## 研究背景与动机
1. **领域现状**：LLM安全对齐主要针对自然语言输入进行防护，但对非自然语言域（数学、代码）的防护存在显著漏洞。
2. **现有痛点**：(a) 白盒攻击需要模型梯度不实用；(b) 黑盒攻击（GCG、AutoDAN）需要多次迭代；(c) 单一域攻击（仅方程或仅代码）成功率有限。
3. **核心矛盾**：LLM在数学推理和代码补全上的强大能力恰好成为绕过安全对齐的弱点——安全训练集中在自然语言上。
4. **本文要解决什么？** 揭示LLM跨域安全漏洞，单次查询即可实现高成功率黑盒越狱。
5. **切入角度**：利用方程和代码两个非自然语言域的协同作用——方程将恶意查询结构化为变量求解，代码将方程包装为功能性补全任务。
6. **核心idea一句话**：方程域+代码域的跨域组合产生协同放大效应，突破自然语言安全对齐的防护。

## 方法详解

### 整体框架
输入：恶意查询。输出：LLM生成的有害回答。Pipeline：(1) 方程模块将查询分解为B（主体）+C（工具）+x（执行步骤）= A（目标）；(2) 代码模块将方程包装在Solver类中要求补全solve()函数。

### 关键设计

1. **方程模块（Equation Module）**：
   - 做什么：将恶意查询结构化为方程求解问题
   - 核心思路：分解为 $B + C + x = A$，其中B是主体、C是工具、x是需要LLM求解的执行步骤
   - 设计动机：LLM的数学推理模式绕过自然语言安全检查，方程形式隐藏了恶意意图

2. **代码模块（Code Module）**：
   - 做什么：用代码包装方程，触发代码补全模式
   - 核心思路：将方程组件封装在Solver类中，设置空的`steps`列表和空的`solve()`方法，要求LLM补全
   - 设计动机：代码补全模式使LLM关注语法正确性而非内容安全性

3. **跨域协同放大**：
   - 做什么：方程+代码的组合效果大于单独效果之和
   - 核心思路：方程提供结构化分解，代码提供功能性包装，两者协同使安全机制更难触发
   - 设计动机：单一域攻击容易被检测，跨域组合创造更强的混淆效果

### 损失函数 / 训练策略
无需训练——纯黑盒攻击，单次查询（no iterative refinement）。520个总查询（AdvBench数据集）。

### 消融实验

| 配置 | 效果 | 说明 |
|------|------|------|
| EquaCode(完整) | 最优 | 方程+代码协同 |
| 仅方程 | 显著低于完整 | 缺少代码包装 |
| 仅代码 | 显著低于完整 | 缺少结构化分解 |
| 协同效应 | > 单独之和 | 证实跨域放大 |

### 关键发现
- **跨域安全漏洞严重**：最新模型（Gemini-1.5/DeepSeek-R1/v3）接近100%被攻破
- **单次查询成功率极高**：无需迭代，520个查询即可系统评估
- **Llama表现异常**：仅17.88%成功率，可能因其对方程格式的理解能力较弱
- 方程和代码的**协同效应>单独效果之和**，证实跨域组合的有效性

## 亮点与洞察
- **暴露了非自然语言域的安全盲区**：揭示了当前LLM安全对齐的结构性弱点。安全训练过度关注自然语言，对数学和代码域防护不足。
- **单次黑盒攻击的高效性**：不需要迭代优化、不需要模型梯度，实际威胁性更大。
- **协同效应的定量证明**：消融实验清晰展示了跨域组合>单域攻击的协同放大。

## 局限性 / 可改进方向
- **防御方案未提出**：仅揭示漏洞但未讨论如何修复
- **Llama异常低成功率**未深入分析原因
- **AdvBench数据集偏小**（520条），且不能代表所有恶意查询类型
- 应进一步研究如何将方程和代码域纳入安全对齐训练

## 相关工作与启发
- **vs GCG/AutoDAN**：这些方法需多次迭代，EquaCode单次即可且成功率更高
- **vs 自然语言越狱（DAN等）**：自然语言越狱依赖角色扮演，EquaCode利用域转换
- 启发：LLM安全对齐需要覆盖所有功能域，不仅是自然语言
