# Thucy: An LLM-based Multi-Agent System for Claim Verification across Relational Databases

**会议**: AAAI 2026  
**arXiv**: [2512.03278](https://arxiv.org/abs/2512.03278)  
**代码**: [GitHub](https://github.com/michaeltheologitis/thucy)  
**领域**: LLM Agent / 事实验证  
**关键词**: 多Agent系统、声明验证、关系数据库、SQL证据生成、跨库推理

## 一句话总结

提出首个跨数据库、跨表的多 Agent 声明验证系统 Thucy，由 Verifier 领导三个专家 Agent（Data/Schema/SQL Expert），对数据源完全无先验知识，能自主发现、推理并生成 SQL 证据，在 TabFact 上超越 SOTA 5.6 个百分点（94.3%）。

## 研究背景与动机

1. **领域现状**：日常生活中充斥着可以用结构化数据验证的声明（如犯罪率下降、经济增长），但因技术门槛高，绝大多数人无法核实。LLM 和 Agent AI 的进步使得自动化验证成为可能。
2. **现有痛点**：当前的事实验证系统存在三大局限——(a) 仅支持单表查询（不能跨表/跨库）；(b) 通常只能处理几百行的小表（需放入 LLM 上下文窗口）；(c) 不提供可追溯的证据（用户无法验证系统的推理）。
3. **核心矛盾**：真实世界的声明验证往往需要跨多个数据库的多表联合查询，且需要对事先未知的数据源进行自主探索。但现有方法假设数据源已知且为单表。
4. **本文要解决什么**：构建一个对数据环境完全无先验知识、支持跨库跨表验证、且能提供具体 SQL 证据的多 Agent 系统。
5. **切入角度**：借鉴古希腊史学家修昔底德（Thucydides）的严格证据标准，设计一个不仅给出判决（验证结果）而且给出证据（SQL 查询及其执行结果）的系统。
6. **核心idea**：将验证任务分解为"数据探索→模式理解→SQL 查询→判决综合"四个阶段，分别由专业化的 Agent 负责，通过 Verifier 协调。关键设计原则是**解耦专家**和**保护 Verifier 上下文**——专家处理脏活累活（探索混乱的数据库），Verifier 只接收精简的高质量信息。

## 方法详解

### 整体框架

Thucy 采用 **Agents-as-Tools** 架构，由一个高层级 **Verifier** Agent 统筹三个专家 Agent：**Data Expert**（概览数据源）、**Schema Expert**（回答 schema 问题）、**SQL Expert**（编写执行 SQL 并返回证据）。三个专家之间不直接通信，只与非 AI 工具交互。Verifier 将专家封装为可调用工具，数据层通过 Google MCP Toolbox 实现即插即用。

### 关键设计

**模块一：Data Expert（数据探索专家）**

- **做什么**：在验证开始前对所有可用数据源进行高层级扫描，产出一段简洁的单段摘要，告诉 Verifier "有哪些数据可发"。
- **核心思路**：订阅 schema 工具集，自主浏览所有连接的数据库，遍历表名、列名、数据类型等元数据，然后将混乱的底层信息凝练为一段清晰的自然语言摘要。
- **设计动机**：数据探索涉及大量底层细节（数据库名、表名、列名、类型、元数据），如果直接暴露给 Verifier 会严重污染其宝贵的上下文窗口。Data Expert"承受这些混乱"，向上只传递精炼信息。

**模块二：Schema Expert（模式理解专家）**

- **做什么**：回答关于数据库模式的任意问题——从列名含义到外键关系、约束条件等。
- **核心思路**：同样订阅 schema 工具集，但被鼓励深入探索底层结构细节。接受两个输入：(1) 自然语言 schema 问题；(2) 上下文提示（如"Seattle, WA"），用于引导 Agent 关注相关数据库。输出格式化的 Markdown 摘要，精确回答 schema 问题。
- **设计动机**：真实数据库通常有不透明的列名（如"NIBRS Group AB"、"Beat"），需要专门的探索和理解。Schema Expert 保护 Verifier 免受混乱的元数据污染，让其以精炼的形式获取准确的 schema 信息。

**模块三：SQL Expert（SQL 查询专家）**

- **做什么**：将自然语言问题转化为 SQL 查询，在数据库上执行，并返回答案及支撑 SQL。
- **核心思路**：接受两个输入：(1) 自然语言数据查询；(2) 相关的 schema 信息（由 Schema Expert 提供）。执行过程模拟真实数据分析师的工作流——先探索数据以理解歧义（如"季度"是日历年还是财年？），再编写查询，处理语法错误和语义不匹配，最终返回答案和对应的 SQL 查询。被指示排除探索性或失败的查询，只保留与最终答案相关的证据。
- **设计动机**：NL2SQL 不是一步完成的过程，需要与数据库的反复交互。SQL Expert 被设计为进入"隧道视野"模式——仅关注当前问题和相关数据片段，避免被无关信息干扰。

**Verifier（验证协调器）**

- **做什么**：协调所有专家 Agent，驱动验证流程，生成最终判决（Verified / Partly Verified / Partly Inaccurate / Inaccurate）和包含解释性 SQL 的分析报告。
- **核心思路**：循环工作流——先询问 Data Expert 获取数据概览 → 咨询 Schema Expert 获取相关 schema → 调用 SQL Expert 查询数据 → 如信息不足则重复循环 → 最终综合生成报告。Verifier **从不直接接触底层数据**，只提出正确的问题并接收精炼回答。
- **设计动机**：Verifier 使用最强力的模型（GPT-5），其上下文需要保持轻量和高信息密度。将"脏活"委托给专家的设计让 Verifier 能专注于高层推理和判断。

### 损失函数 / 训练策略

本文不涉及模型训练。系统基于 OpenAI Agents SDK 构建。Verifier 使用 GPT-5，专家 Agent 使用 GPT-5-mini（或降级到 GPT-4o-mini）。数据层通过 Google MCP Toolbox 连接，支持 PostgreSQL、MySQL 等。专家 Agent 的记忆仅在单次工具调用内持续（不跨调用），保持原子性和可复用性。

## 实验关键数据

### 主实验

TabFact benchmark（小测试集，~2k 条事实-表对）的准确率对比：

| 方法 | 模型 | 准确率 (%) |
|------|------|-----------|
| BINDER | Codex | 85.1 |
| DATER | Codex | 85.6 |
| CoTable | PaLM 2 | 86.6 |
| ReAcTable | Codex | 86.1 |
| AutoTQA | GPT-4-turbo | 88.7 |
| POS | GPT-4o-mini | 82.7 |
| **Thucy** | GPT-4o-mini | **93.7** |
| **Thucy** | GPT-5-mini | **94.3** |

### 消融实验

- **模型降级鲁棒性**：专家 Agent 从 GPT-5-mini 降级到 GPT-4o-mini 后，准确率仅下降 0.6%（94.3% → 93.7%），仍然大幅领先 SOTA（+5.0%）。说明架构设计比单个 Agent 的模型能力更重要。
- **跨库跨表能力**：Thucy 是唯一同时支持 Cross-Table + Cross-Database + Source-Agnostic + Verifiable 的系统。

### 关键发现

1. **专家化分解效果显著**：即使用较弱模型（GPT-4o-mini），Thucy 仍大幅超越用同等或更强模型的 baseline。
2. **Source-Agnostic 设计可行**：系统无需预先了解任何数据库信息，完全自主发现和探索。
3. **SQL 证据的价值**：在西雅图犯罪率验证案例中，Thucy 不仅给出判决，还返回可直接执行的 SQL 查询，用户可修改查询进一步探索。
4. **真实案例验证**：对西雅图市检察官关于"2024 年犯罪率下降"的声明，Thucy 发现实际数据显示犯罪率略有上升（财产犯罪 +0.7%，暴力犯罪 +0.8%），判定声明"不准确"。

## 亮点与洞察

- **"Agent-as-Tools"模式**的干净实现：专家封装为工具、原子化执行（无跨调用记忆）、Verifier 上下文保护——三个设计共同构成了可扩展的多 Agent 架构。
- **MCP Toolbox 的工程价值**：通过 YAML 配置即可插拔数据库，展示了 MCP 标准化协议在实际 Agent 系统中的实用性。
- **真实新闻验证案例**极具说服力：修昔底德式的严格证据标准在现代信息验证中的复兴。对西雅图犯罪率和"暴力犯罪骤降 36%"两个案例的层层追查，展示了系统在模糊定义（如"downtown"的不同含义）面前的处理能力。
- **Schema Expert 的必要性**：真实数据库中不透明的列名和复杂的 schema 是 NL2SQL 的主要障碍，专门的 schema 理解 Agent 是关键贡献。

## 局限性 / 可改进方向

1. **仅评估单表 benchmark**：TabFact 本身是单表数据集，未充分展示 Thucy 的跨库跨表优势（仅通过案例研究定性展示）。
2. **成本较高**：Verifier 使用 GPT-5，三个专家各自执行大量工具调用，实际部署成本值得关注。
3. **依赖 SQL 数据源**：无法处理非关系型数据（如 JSON、CSV、图数据库），也无法验证需要非结构化信息（如文本、图片）的声明。
4. **声明需手动提交**：未集成自动声明提取（如从新闻文章中自动抽取可验证声明）。
5. **错误传播风险**：专家 Agent 的错误（如 schema 理解错误或 SQL 语义错误）可能传播到 Verifier。

## 相关工作与启发

- **事实验证系统**：BINDER（NL→程序生成）、DATER（分解+聚合）、CoTable（多步推理）、ReAcTable（ReAct 模式）——都限于单表。
- **AutoTQA**：首个支持跨表的系统，但使用循环编排模式（执行→批评→精化循环），且不提供可追溯证据。
- **POS**：关注可解释性（返回原子步骤序列），但限于单查询且不可直接验证。
- **MCP 协议**：Anthropic 提出的标准化 AI-数据源连接协议，已被 Paypal、Databricks、Snowflake 等采用。
- **启发**：多 Agent 系统的成功关键不在单个 Agent 的能力上限，而在于合理的任务分解和上下文管理策略。

## 评分

⭐⭐⭐⭐

系统设计精良——Source-Agnostic + 跨库跨表 + SQL 证据的组合在事实验证领域是首创。Agents-as-Tools 的架构设计干净务实，真实案例极具说服力。在 TabFact 上的 SOTA 结果令人信服。主要不足是 benchmark 未能充分展示系统的核心优势（跨库跨表），且部署成本较高。
