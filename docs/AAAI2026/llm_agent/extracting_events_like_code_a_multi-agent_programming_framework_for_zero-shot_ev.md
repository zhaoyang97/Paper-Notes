# Extracting Events Like Code: A Multi-Agent Programming Framework for Zero-Shot Event Extraction

**会议**: AAAI 2026  
**arXiv**: [2511.13118](https://arxiv.org/abs/2511.13118)  
**代码**: https://github.com/UESTC-GQJ/Agent-Event-Coder  
**领域**: NLP / 事件抽取  
**关键词**: 零样本事件抽取, 多智能体, 程序化框架, Python类模板, 双循环细化

## 一句话总结

提出 Agent-Event-Coder (AEC)，将零样本事件抽取类比为软件工程流程，用4个专职Agent（Retrieval→Planning→Coding→Verification）协作完成抽取，并将事件schema编码为可执行Python类实现编译器式确定性验证与双循环迭代修正，在5个领域、6个LLM上全面超越零样本基线。

## 研究背景与动机

1. **事件抽取的重要性**：事件抽取（EE）从非结构化文本中识别事件触发词及其论元，在知识库构建、信息检索、问答系统中有核心作用。
2. **监督方法的瓶颈**：传统EE依赖大量标注样本，但事件类型持续增长且标注成本极高，无法为每个新类型都收集训练数据。
3. **零样本EE的潜力与挑战**：ZSEE仅用事件类型名或定义即可抽取未见类型，扩展性强，但实际效果远未令人满意。
4. **上下文歧义问题**：触发词常有多义性（如"strike"可指罢工或攻击），零样本设定下LLM容易过度依赖触发词本身而忽略上下文线索，导致类型误判。
5. **结构保真问题**：EE要求输出严格符合预定义schema（如JSON），但LLM在未微调时经常生成违反schema约束的输出——包含不存在的角色、幻觉论元、数据类型错误等。
6. **现有方法的不足**：单Agent直接prompting方法（如DirectEE、ChatIE）对prompt设计高度敏感，缺乏显式分解，无法处理触发词-论元间的复杂交互；已有多Agent IE工作未同时解决语义消歧和结构验证。

## 方法详解

### 整体框架

AEC将ZSEE重新定义为"协作式可验证代码生成"过程。输入为非结构化文本 $T$ 和未见事件schema $S_e = \langle e, R_e \rangle$（$e$为事件类型，$R_e$为论元角色及其类型），输出为事件实例 $y = \langle e, z, A \rangle$（$z$为触发词，$A$为论元-角色对集合）。整个pipeline由4个专职Agent串联，配合内外双循环反馈机制实现迭代推理与验证。

### 关键设计

#### 1. Retrieval Agent（检索代理）

- **做什么**：针对给定schema $S_e$，自动生成 $k$ 条高质量示例句子 $D_{ex} = \{s_1, \dots, s_k\}$，作为schema到文本的"类比桥梁"。
- **核心思路**：借鉴类比prompting范式（Yasunaga et al. 2024），让LLM自行构造与schema语义对齐的示例，而非依赖外部检索。
- **设计动机**：零样本设定下无标注样本可用，自生成示例能将抽象的schema约束映射为具体的语言实现，帮助后续Agent降低多义触发词的早期承诺错误。

#### 2. Planning Agent（规划代理）

- **做什么**：结合输入文本 $T$、schema $S_e$ 和检索到的示例 $D_{ex}$，生成 $k$ 个排序的触发词-类型假设列表：$P = \{((z_i, e), \beta_i, \rho_i)\}_{i=1}^k$，其中 $\beta_i \in [0,1]$ 为置信度，$\rho_i$ 为自然语言理由。
- **核心思路**：综合利用词汇和语义线索，为每个候选触发词提供"为什么它可能激发该事件类型"的解释性推理。
- **设计动机**：生成多个ranked假设而非单一预测，为后续验证失败时的回退提供候选池；保留推理理由 $\rho_i$ 便于错误分析和可解释性。

#### 3. Coding Agent（编码代理）

- **做什么**：将最高置信度假设 $((z^\star, e), \beta^\star, \rho^\star)$ 转化为可执行Python代码，实例化预定义的事件类（基于Pydantic BaseModel）。
- **核心思路**：将事件schema编译为Python类定义，schema合规性等价于构造一个合法的类实例——这可以在运行时被确定性验证。
- **设计动机**：利用编程语言的类型系统和运行时验证能力，将"结构保真"问题从自然语言约束转化为编程约束，使验证从概率性变为确定性。

#### 4. Verification Agent（验证代理）

- **做什么**：对生成的代码对象 $C_{obj}$ 执行三阶段测试套件，返回二值判定 $V$ 和诊断信息 $\varepsilon$。
- **三阶段验证**：
  - **语义检查 $\mathcal{T}_1$**：验证预测的触发词 $z^\star$ 确实出现在输入文本中且与事件类型语义兼容。
  - **类型检查 $\mathcal{T}_2$**：验证每个论元值符合schema规定的数据类型，多值约束通过Pydantic验证。
  - **结构检查 $\mathcal{T}_3$**：确认代码可编译、包含exacty三个字段（event_type, trigger, arguments）、且能序列化。
- **设计动机**：用编译器式的确定性反馈替代模糊的自然语言反馈，验证失败时提供精确的错误定位，指导下一轮修正。

#### 5. 双循环细化算法

- **做什么**：当验证失败时执行两层迭代——内循环在当前假设上最多尝试 $t$ 次patch修正；若内循环耗尽仍未通过，外循环回退到下一个置信度最高的假设重新编码。
- **核心思路**：遍历 $O(kt)$ 条候选路径，保证最终输出同时满足语义正确性和结构一致性。
- **设计动机**：单次生成难以保证质量，多假设+多尝试的双循环架构既提高了召回率也保障了结构保真，且在 $k=t=3$ 时效果已饱和、计算开销可控。

## 关键发现

1. **AEC在所有5个领域、6个LLM上均为最优**，尤其在schema复杂的数据集（ACE、CASIE）上优势更显著，ACE TC上比最强基线ChatIE高出+7.1%（70B）。
2. **验证循环的贡献最大**：去掉Verification Loop后性能下降最剧烈（ACE TC: 54.6→44.7，跌幅-9.9），说明编译器式验证是AEC的核心驱动力。
3. **Retrieval Agent对消歧至关重要**：移除后FewEvent TI下降-5.6，自生成示例有效弥补了零样本缺乏上下文参照的不足。
4. **假设数 $k$ 和patch次数 $t$ 在 $k=t=3$ 时饱和**：进一步增大带来的收益极小（GPT-4o上ACE TC: 61.8→62.3），但计算成本线性增长。
5. **模型规模效应明显**：GPT-4o > Qwen2.5-72B > Llama3-70B > 小模型，AEC能从更强的backbone中稳定获益。

## 亮点与洞察

- **"像写代码一样抽取事件"**的类比非常精妙：将schema表示为Python类、将验证等价于单元测试、将修正等价于debug，建立了NLP结构化预测与软件工程之间的桥梁。
- **确定性验证 vs 概率性评估**是本文最核心的设计哲学：自然语言反馈是模糊的，而类型系统+编译器诊断是确定的，后者从根本上解决了LLM输出的结构保真问题。
- **双循环回退机制**优雅地处理了"第一个假设可能错"的问题，保留多假设的探索空间同时不牺牲效率（$O(kt)$复杂度）。
- 定性分析表明各Agent的关注点正交互补：Planning关注触发词选择，Coding补充论元填充，Verification纠正类型错误——三者形成渐进式质量提升。

## 局限性 / 可改进方向

1. **计算开销**：4个Agent顺序执行+双循环迭代，每条样本需要多次LLM调用，推理效率低于单步方法，难以大规模部署。
2. **Agent间通信为串行**：当前架构中Agent之间是线性流水线，没有利用并行或异步的可能性。
3. **自生成示例的质量无保证**：Retrieval Agent的示例是LLM自行生成的，可能引入偏差或幻觉，尤其在高度专业化领域（如生物医学）。
4. **仅评估了英文数据集**：跨语言泛化能力未知。
5. **事件重叠/嵌套场景未讨论**：一个句子包含多个互相关联事件时，当前pipeline的处理方式不清晰。
6. **$k=t=3$ 的超参数设定**：虽然实验表明饱和，但不同领域/模型的最优值可能不同，缺乏自适应调节机制。

## 相关工作与启发

- **Code4Struct / GuidelineEE**：将事件抽取表示为代码生成的先驱工作，AEC在此基础上引入多Agent协作和确定性验证。
- **ChatIE**：多轮对话式抽取，AEC用结构化Agent流水线替代了非结构化对话。
- **CMAS (NER)**：最接近的多Agent IE工作，但专注于NER且无代码验证机制。
- **启发**：该"like code"范式可推广到其他结构化预测任务（如关系抽取、槽填充），核心思路是用编程语言的类型系统做确定性输出约束。多Agent+代码验证的架构模式对Agent-based IE是一个值得follow的方向。
